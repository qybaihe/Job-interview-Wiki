# TODO_DEEPENING_BACKLOG.md

## 目标

在 `Job-interview-Wiki` 已完成第一轮铺面后，建立一个**可持续消耗、可被 cron 长期推进、且质量优先**的深度待办池。

这份 backlog 的作用不是简单列更多公司名，而是把后续工作拆成几类：

1. **从零建设**：目录已规划但尚未有真实题库/面经
2. **薄覆盖补厚**：已有 1 份面经或只有样板，优先补真实内容
3. **强方向深挖**：已有 2~4 份面经，继续做角色细分 / 场景细分 / 社招校招分层
4. **资产清洗**：清理样板、命名漂移、README 计数过时、去重与结构统一
5. **研究型补强**：针对高价值方向补专题题库，而不是重复堆 JSON

---

## 当前仓库状态（2026-03-20 盘点）

### 结构面观察
- 已覆盖 **16 家公司**
- 实际已出现约 **50 个方向目录**
- 已有 **41 份题库**
- 已有 **130 份结构化面经 JSON**
- 部分 README / 状态文件仍停留在旧计数（例如 README 仍写 113 份 JSON）

### 现阶段的主要矛盾

仓库已经不再缺“有没有内容”，而是开始缺：

1. **优先级管理**：哪些方向继续补最值钱
2. **覆盖均衡**：有的方向 7~11 份 JSON，有的方向只有 1 份或仅样板
3. **真实性门槛**：cron 不能再靠“读目录 + 更新状态”伪装完成
4. **资产治理**：样板文件、旧 README 统计、命名不统一会拖低可信度
5. **可持续性**：必须保证 cron 总有高质量任务，而不是做完 20 条就枯竭

### 后续策略

后续 backlog 统一遵循：
- **先做薄弱但高价值方向**，再做锦上添花
- **先补真实公开面经**，再做题库抽象
- **优先补“有招聘热度 + 有业务辨识度 + 有公开来源”的方向**
- **每轮 cron 最好只吃 1~2 个任务，避免发散和造垃圾**

# 六、建议给 cron 的消费规则

## 每轮优先级
1. **优先提交工作区已有真实产出**
2. 再做 **P0 从零建设**
3. 再做 **P1 薄覆盖补厚**
4. 再做 **P2 强方向深挖**
5. 如果搜索不到可信来源，则消费 **P3 资产治理**

## 每轮完成判定
只有满足以下之一才能算成功：
- 新增 1 份真实面经 JSON
- 或对 1 个题库新增 >= 15 条可溯源且非重复的问题
- 或完成 1 项明确的资产治理（README 统计修正 / 样板清理 / 命名统一）并提交

## 不算成功的情况
- 只读目录
- 只写 progress_state.json
- 只说“已检查结构”
- 只创建空目录 / 样板文件
- 只有样板、没有真实来源内容

## 结论

仓库现在最需要的不是“更多随机内容”，而是：
- 一份**足够深、足够长、可以持续被 cron 消耗**的任务池
- 一套**优先补薄弱方向、搜索失败就做资产治理**的退路机制
- 一条**禁止空转、禁止假完成**的执行规则

这份 backlog 就是给 cron 的长期粮仓。

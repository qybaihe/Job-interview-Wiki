# merge_notes.md

## 第二阶段 clean_merge 记录

目标：
- 避免扩张阶段把同一公司/部门反复记成多条 completed 记录
- 让 `progress_state.json` 既能做统计，也能直接给 report/orchestrate 消费
- 把“题库样板 / 面经样板 / 后续补来源动作”分层记录清楚

---

## 已执行的合并规则

### 1. completed 按公司 / 事业群 / 部门聚合
- 不再因为新增一个岗位样板，就额外生成一条完全重复的 completed 记录
- 同一部门下的多个产物统一挂到：
  - `artifacts[]`
  - `question_bank_ready`
  - `interview_json_count`

### 2. 岗位扩张不单独重复记基础条目
例如：
- 腾讯 / 微信事业群 / 基础平台
- 美团 / 到店事业群 / 后端技术
- 拼多多 / 主站业务 / 服务端

这些条目已经有后端样板后，后续新增算法/客户端样板时：
- 优先更新已有 completed 项
- 不再堆重复 completed 条目

### 3. next_actions 只保留真正下一步动作
当前阶段优先是：
- `clean_merge`
- `source_enrich`
- 少量继续补结构化 JSON

不再把已经落完题库+样板的目标继续重复列入“新增内容”动作。

---

## 后续 source_enrich 规则

在没有稳定真实公开面经链接前：
- 可以保留 `manual-curated` / `manual-summary` 类型
- 但 note 必须明确“这是样板，不是已核验真实面经原帖”
- 一旦补到真实公开来源：
  1. 优先更新 JSON 的 `source`
  2. 再视情况把高频题反向回写题库
  3. 不重复新建同义样板文件

---

## 不再做的事
- 不再把同一个部门因多个岗位样板反复计入 completed
- 不再让 orchestrate 每次把 progress_state 刷回更早统计口径后不收口
- 不再只改状态数字、不改真实内容

## 基础知识
- 多线程场景里，IO 线程越多越好吗？为什么？
- 操作系统中一次 IO 的全过程是什么？
- 虚拟内存怎么理解？
- 从输入 URL 到页面显示的全过程是什么？
- IO 多路复用是什么？select / poll / epoll 有什么区别？
- Redis 常见数据结构及其时间复杂度是什么？
- 海量 IP 地址存储在磁盘中，如何判断某个 IP 是否存在？
- 介绍一下数据库索引。
- 哪些情况下会发生索引失效？`IN` 会失效吗？`LIKE` 呢？
- MySQL 默认隔离级别是什么？它是如何实现的？
- 如果大量请求访问的资源在 Redis 中不存在，会出现什么问题？怎么解决？
- 类加载器机制说一说。
- HashMap 说一说；HashMap 和 ConcurrentHashMap 有什么区别？
- volatile 的语义是什么？如何保证可见性和禁止重排序？
- Java 常见 GC 算法有哪些？Minor GC 和 Full GC 区别是什么？
- 乐观锁和悲观锁分别适合哪些场景？

## 业务场景
- 如果 Redis 故障，飞书这类协同系统主流程如何降级？
- 如果做飞书消息投递链路，如何处理削峰、幂等、重试和多端同步？
- 如果做多租户权限系统，如何在隔离性和查询性能之间取舍？
- 如果做协同文档编辑后台，如何在一致性、延迟和可用性之间权衡？
- 飞书这类办公协同产品为什么会特别关注租户隔离、权限边界和消息可靠性？
- 飞书这类企业 IM，如果同一用户在手机、桌面端、Web 端同时在线，消息投递和未读态如何设计，才能既保证多端同步又控制写放大？
- 如果消息发送成功但回执丢失，客户端和服务端如何做幂等补偿，避免重复展示或未读数错误？
- 企业 IM 的历史消息拉取为什么常把冷热分层存储和最近会话缓存拆开设计？
- 群聊消息洪峰下，如何在写扩散和读扩散两种收件箱模型之间取舍？
- 已读回执、撤回、编辑、引用回复这些能力叠加后，消息模型应该如何设计版本和状态字段？
- 飞书文档、表格、知识库这类协同产品，权限模型为什么通常不止 RBAC，还会叠加资源继承、分享链路和租户边界？
- 如果一个文档既支持组织内分享，又支持跨租户邀请，后端如何防止权限穿透和缓存脏读？
- 协同编辑后台里，哪些数据适合走强一致事务，哪些更适合事件驱动的最终一致？为什么？
- 多租户 SaaS 场景下，数据库层按库隔离、表隔离、行隔离分别适合什么阶段？飞书这类产品可能怎么权衡？
- 如果租户管理员批量调整组织架构和角色权限，如何避免权限缓存大面积失效带来的瞬时流量抖动？
- 对消息投递或文档变更通知这种高频异步链路，MQ 的重试、死信、顺序性和幂等键应如何联合设计？
- 如果一个租户短时间内触发超大规模机器人通知或消息风暴，如何做租户级限流而不伤害全局其他租户？
- 飞书这种 toB 协同产品为什么比消费互联网更强调审计日志、操作留痕、可追溯回放？后端怎么落地？
- 当某个下游组织服务或权限服务高延迟时，消息和文档主链路如何做超时、熔断、降级和数据兜底？
- 如果要给飞书开放平台设计 webhook / callback 事件投递能力，怎样处理签名校验、重放攻击、重试退避和订阅隔离？
- 如果让你从零设计一个飞书内部后台服务，除了存储和缓存，还应优先考虑哪些平台化能力：灰度、审计、权限、配额、监控、告警、容灾？

## 系统设计
- 设计一个企业 IM 消息存储与投递系统，如何处理已读未读、多端同步和历史消息拉取？
- 设计一个多租户后台权限系统，如何支持组织、部门、角色和资源级授权？
- 设计一个文档协同编辑后台时，哪些能力适合强一致，哪些能力适合最终一致？
- 如果消息系统面临突发洪峰，如何做削峰、重试、幂等与死信处理？
- 如果需要支持文档协同编辑后台，哪些环节更适合强一致，哪些环节可以接受最终一致？
- 海量 IP 地址持久化在磁盘中，如何快速判断某个 IP 是否存在？如果继续扩大到去重/计数，又该怎么设计？

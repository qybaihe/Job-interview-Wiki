# 顺丰｜物流科技 / 后端研发｜Question Bank（partial）

## 说明

- 本题库聚焦 **顺丰（含顺丰科技 / 科技服务 / 物流科技场景）后端研发**。
- 当前轮次目标优先完成可复习题库；由于公开可直接核验的**单篇顺丰后端面经原帖较少且搜索接口受限**，本稿采用：
  - **公开业务/产品信息**（顺丰官网、科技服务产品页可见信息）
  - **可见搜索结果页中的岗位/面试关键词**
  - **物流科技后端岗位通用高频考点**
- 因此本稿标记为 **partial / no-source mixed**：
  - 能映射到公开业务事实的问题，标注公开来源；
  - 无法映射到单一公开面经原帖的问题，保守标注 `no-source`，仅作为候选高频题，不冒充已核验单帖。

## 公开业务基底

可从顺丰官网看到的业务/技术线索包括：

- 快递时效产品：顺丰特快、顺丰标快、半日达、同城急送
- 供应链/仓配：仓储服务、供应链仓配一体、冷运仓、医药仓、国际仓配
- 科技服务 / SaaS：顺丰快递管家、函证通、销果裹、快寄顺丰、数智太白
- 供应链软件：丰智云链、丰智云策、丰智云塔、丰智云商、丰智云数
- 智能硬件：丰翼无人机、顺丰无人车、智能柜
- 行业解决方案：高科技、工业设备、消费品、零售食品、生命科学与医药、金融保险、政企政务

> 这意味着顺丰后端研发题目，大概率不会只停留在通用 CRUD，而会与 **履约链路、时效、路由、运价、仓配一体、设备接入、B 端 SaaS、多租户、稳定性** 等业务耦合。

## 一、物流履约链路与领域建模

### 1. 你如何设计“寄件 -> 揽收 -> 中转 -> 派送 -> 签收”全链路状态机？
- **考察点**：订单状态流转、幂等、补偿、异常件处理
- **业务映射**：顺丰官网公开呈现多种快递/同城/大件服务，后端必然要支撑多环节履约状态管理
- **来源**：<http://www.sf-express.com/chn/sc/>

### 2. 如果同一票快件同时收到多个扫描/轨迹事件，如何保证轨迹写入顺序与最终一致性？
- **考察点**：消息乱序、事件时间 vs 处理时间、版本号/CAS、去重
- **业务映射**：物流轨迹天然是事件流系统
- **来源**：公开业务事实可映射；具体问法未见单帖，记为 `partial`

### 3. 快递订单和运单应该如何拆分建模？什么时候一单一运单，什么时候一单多包裹？
- **考察点**：领域建模、主子单设计、包裹粒度、扩展性
- **业务映射**：大件、仓配、国际与普通快递在履约粒度上可能不同
- **来源**：顺丰官网产品线可映射；具体问法 `no-source`

### 4. 异常件（拒收、破损、丢件、地址异常）在系统里如何建模，如何避免状态互相覆盖？
- **考察点**：异常域建模、状态机保护、审计日志
- **来源**：物流业务通用高频；`no-source`

## 二、高并发写入、消息队列与幂等

- **业务映射**：物流节点扫描、轨迹写入是顺丰类场景典型高并发入口

- **来源**：`no-source`

### 8. 如何设计“失败可重试但不重复扣费/不重复派单”的异步任务系统？

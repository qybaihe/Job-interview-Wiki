# 科大讯飞｜大模型平台｜推理工程｜question-bank

> 说明：本题库基于科大讯飞 / 星火官方公开文档做岗位能力映射整理。
> 当前更适合作为正式 question-bank 入口，不伪称为单场真实面试逐字题单。
> 本次增强重点补足：错误码治理、版本演进、搜索增强、跨语言 SDK、一致性鉴权、容量隔离与可观测性设计。

## 适用岗位
- 大模型平台后端
- 推理工程 / 在线推理平台
- 模型网关 / 流式服务治理
- 平台后端（偏 LLM 接入与多模型路由）

## 一、接入协议与鉴权实现

1. WebSocket 鉴权里为什么要求客户端自己生成带签名的 URL，而不是简单传一个 Bearer Token？
2. `date` 参数为什么必须采用 RFC1123 格式，且时间偏差必须控制在 300 秒内？如果线上机器时钟漂移，会出现什么故障特征？
3. 为什么签名串明确要求包含 `host`、`date`、`request-line` 三段，而不是只签 path 或时间戳？
4. `authorization_origin` 里已经包含签名信息，为什么最终还要再次 base64 编码成 `authorization`？服务端通常怎么校验这一步？
5. 如果你来写跨语言 SDK，怎样避免 Python / Java / Go 在时间格式、换行、空格、URL 编码上出现签名不一致？
6. 为什么官方文档特别强调“不同 domain 取值对应的 URL 也不一样”？如果客户端只改了 domain，没有同步改 URL，会带来什么问题？
7. SDK 文档提到要做统一连接池、httpDNS、上下文绑定和排障日志回传，这些能力分别解决什么线上问题？
8. 如果偶发出现“鉴权偶尔失败，但重试又成功”，你会先排查哪些点：时钟、代理改写 host、编码差异、签名串拼接、网关时间窗口还是 DNS？

## 二、流式返回、会话状态与前端协作

9. `header.status` 和 `payload.choices.status` 都有 `0/1/2`，它们分别描述什么状态？为什么不能只看最后一个包？
10. 官方文档要求“开发者需要将接口多次返回的结果进行拼接展示”，如果拼接逻辑写错，常见线上问题会有哪些？
11. 如果要让用户感觉“回复来得快”，前端更适合按 token 刷新、按短句刷新，还是按标点刷新？为什么？
12. 当 `show_ref_label=true` 时，为什么系统会先返回检索信源，再返回模型正文？这种设计对前端渲染和可解释性分别意味着什么？
13. 如果流式过程中网络断开，且服务端已经产出了一半内容，你会如何定义客户端重试语义：整轮重试、幂等恢复、还是禁止自动重试？
14. 审计日志应该记录首帧、全量 chunk、聚合后的最终文本，还是三者都要？不同粒度分别服务什么场景？
15. 为什么平台必须同时观测 TTFT、总完成时长、连接占用时长和 tokens/s，而不能只看单一 RT？
16. 用户反馈“首字很快，但后面越来越慢”，你会如何区分是模型解码、网络 flush、连接拥塞还是下游消费背压的问题？

## 三、模型版本、路由与能力抽象

17. 从 Lite / Pro / Pro-128K / Ultra / kjwx 这些版本命名里，你能看出平台做了怎样的能力分层？
18. 为什么平台更倾向让业务方传 `domain`，而不是暴露底层物理集群地址或具体模型实例名？
19. `kjwx` 这种垂直领域模型接入通用平台时，网关层应该如何做能力发现、参数约束和兜底回退？
20. Max 版本下线并并入 Ultra，对 SDK、模型路由、配额和灰度发布分别意味着哪些改造？
21. 当一个平台同时支持通用对话、长上下文、垂直模型和联网搜索能力时，如何设计统一但不过度耦合的请求协议？
22. 如果某业务强行把所有请求都打到 Ultra，你会如何用线上数据证明需要分层路由，而不是继续“全量高配”？
23. 对预算敏感但偶尔需要长上下文的业务，如何在 Pro / Pro-128K / Ultra 之间做自动路由和降级？
24. 如果你要做多模型路由，哪些维度最值得纳入决策：上下文长度、SLA、搜索需求、垂类需求、预算、时段流量还是用户等级？

## 四、token、上下文与成本治理

25. 官方文档给出“1 token 约等于 1.5 个中文汉字或 0.8 个英文单词”，为什么线上限流或计费不能直接按字符数做？
26. `question_tokens`、`prompt_tokens`、`completion_tokens`、`total_tokens` 分别更适合支撑哪些平台决策？
27. 为什么不同模型版本要独立计量 tokens？如果账单层把不同模型混算，会带来哪些问题？
28. 当历史上下文越来越长时，为什么它不只影响成本，还会伤害首 token 延迟和整体尾延迟？
29. 如果一个租户频繁传“超长历史 + 超长生成”，你会如何识别、限额并给出产品化的优化建议？
30. 当 `search_mode=deep` 时为什么会出现额外 token 消耗？这对预算管理和配额预测意味着什么？
31. 如果你要实现“自动裁剪历史消息”，会优先保留哪些消息，删除哪些消息？为什么不能简单按时间截断？
32. 平台如何识别“问题本身不贵，但历史上下文过长导致成本失控”的请求模式？

## 五、错误码治理、重试与容量保护

33. `10006 用户并发错误` 背后可能对应怎样的连接状态管理设计？为什么平台要限制同一用户多处同时连接？
34. `10007 用户流量受限` 为什么特别强调“必须等大模型完全回复后，才能发送下一个问题”？这暴露了什么会话模型？
35. `10008 服务容量不足` 与 `10110 服务忙` 在排障上有什么区别？分别更像系统性容量问题还是瞬时拥塞？
36. `10907 token数量超过上限` 为什么不应该靠盲目重试解决？更合理的产品化处理方式是什么？
37. `11200/11201/11202/11203` 这组授权 / 流控错误，分别应映射到怎样的告警、重试和用户提示策略？
38. `10222 引擎网络异常` 和 `10012 引擎内部错误` 分别更可能落在哪一层？你会如何分治排障？
39. 输入审核不通过 `10013` 和输出审核不通过 `10014`，在用户提示、审计留痕和前端展示上为什么应该区别处理？
40. 为什么流式大模型服务不能只依赖 HTTP 状态码表达异常，而往往还要保留细粒度业务错误码？

## 六、搜索增强、插件能力与可信性

41. 为什么只有 Pro / Ultra 系列支持联网搜索插件？这说明搜索增强对模型和平台资源提出了什么额外要求？
42. `search_mode=normal` 与 `search_mode=deep` 的根本差异是什么？如果 deep 更贵更慢，你会如何做 A/B 评估？
43. `show_ref_label=true` 时为什么要先返回信源，再返回模型正文？如果统一 SDK 同时兼容“有信源 / 无信源”两类返回，输出抽象怎么设计？
44. 如果搜索结果与模型固有知识冲突，平台层是否应该做信源排序、引用限制、安全兜底或提示降置信？为什么？
45. 哪些问题最值得触发搜索插件，哪些问题则更适合直接靠模型内部知识作答？你会怎样做策略判断？
46. 如果业务方抱怨“搜索后更贵更慢，但回答没明显变好”，你会从哪些维度验证是策略问题、索引问题、提示词问题还是评估口径问题？

## 七、系统设计题

47. 设计一个面向外部开发者的大模型网关，支持 WebSocket 鉴权、流式返回、联网搜索、版本路由、错误码治理和租户配额控制。
48. 设计一套跨语言 SDK，要求支持鉴权 URL 生成、连接复用、流式拼接、sid 留存、错误分类和多模型切换。
49. 设计一套推理平台可观测体系，至少覆盖鉴权成功率、TTFT、tokens/s、错误码分布、连接占用时长、模型版本、租户维度和搜索开关维度统计。
50. 设计一套多模型路由系统：根据上下文长度、预算、SLA、是否需要搜索、是否为垂直领域请求自动选择模型并支持快速降级。
51. 设计一套 token 计量与账单系统，要求不同模型版本独立计量，同时支持 search_mode=deep 带来的额外消耗归因。
52. 如果新版本灰度后 `10110` 暴涨但整体 QPS 没变，你会如何判断是灰度策略失衡、路由失衡、实例容量下降、连接泄漏，还是请求结构发生变化？

## 八、手撕 / 实作追问

53. 手写一个鉴权 URL 生成函数，输入 host / path / apiKey / apiSecret / date，输出最终 WebSocket URL，并指出最容易写错的细节。
54. 给你一串流式 chunk，如何在保证展示稳定性的前提下做增量拼接，并兼容“信源先到、正文后到”的混合流？
55. 实现一个基于错误码的 retry policy 映射表，避免把重试逻辑写成散落的 if-else，你会怎么设计结构？
56. 如果 sid 是线上排障关键字段，你会把它打到哪些日志、指标、trace 和告警上下文中？

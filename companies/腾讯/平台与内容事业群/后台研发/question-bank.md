> 当前定位：基于大厂后台研发通用高频题、腾讯系常见工程化要求以及内容平台业务特点做的结构化题库，用于后续继续补真实面经与反向生成结构化 JSON。

## 1. 业务理解与项目题

### 1.1 内容平台项目拆解
- 讲一个你做过的高并发后台项目，核心业务链路、峰值流量和性能瓶颈分别是什么？
- 如果是内容分发、互动、消息、评论、推荐召回等场景，你在系统里承担的具体模块是什么？
- 项目里最难的一次线上事故或稳定性问题是什么？你怎么止损、复盘、避免再次发生？
- 如果业务要求新增一个实时互动功能，你会怎么评估接口、存储、消息链路和容量风险？

### 1.2 指标与业务抽象
- 内容平台常见的核心指标有哪些？播放、停留时长、互动率、转化率分别会影响哪些后端设计？
- 如果业务增长很快，但后端资源成本也同步暴涨，你会从哪些角度做优化？
- 后台研发为什么不能只盯接口 QPS，还要看成功率、延迟分位、消息堆积和回放能力？

## 2. Java / Go / 基础语言能力

### 2.1 Java 基础
- `HashMap` 为什么线程不安全？在并发场景下会出什么问题？
- `ConcurrentHashMap` 在不同 JDK 版本的实现差异是什么？
- 说一下 Java 内存模型、`volatile`、可见性和有序性。
- 什么情况下会触发 Full GC？如何排查频繁 GC？
- 线上 OOM 你一般怎么定位？会看哪些指标和 dump 信息？

### 2.2 Go 基础
- goroutine 和线程的区别是什么？为什么 Go 适合高并发服务？
- channel 适合解决什么问题？它的误用会带来哪些风险？
- Go 中内存逃逸、GC 和对象分配如何影响接口性能？
- 线上 Go 服务延迟抖动明显时，你会怎么排查调度、锁竞争和网络问题？

## 3. 数据库与存储

### 3.1 MySQL
- MySQL 索引失效的典型情况有哪些？
- 联合索引最左匹配原则怎么理解？真实项目里怎么设计复合索引？
- 深分页为什么慢？如何优化？
- 事务隔离级别分别解决了什么问题？MVCC 是怎么实现的？
- 如果热点行更新冲突严重，你会怎么处理？

### 3.2 Redis
- Redis 为什么快？单线程模型的边界在哪里？
- 缓存穿透、缓存击穿、缓存雪崩分别怎么治理？
- 如何设计一个热点内容缓存方案，既保证命中率又避免脏数据长期滞留？
- Redis 分布式锁为什么容易踩坑？常见问题有哪些？

### 3.3 消息与日志链路
- Kafka 为什么适合做异步削峰？它和 RocketMQ 在关键设计上有什么差异？
- 如何保证消息不丢、不重、可追踪？
- 如果内容审核链路是异步多阶段处理，如何设计消息重试、死信和补偿机制？

### 4.1 分布式基础

### 4.2 高并发设计
- 如何设计一个支持高并发评论 / 点赞 / 弹幕的后台系统？
- 秒杀、热点活动、直播互动等流量尖峰场景下，哪些环节最容易成为瓶颈？

### 4.3 一致性与可用性取舍

## 9. 清洗版补充：内容平台高并发与回放治理框架

> 说明：这一节把内容平台后台研发里容易分散出现的评论、点赞、消息、审核、回放、容灾问题收敛成更适合真实后端面试回答的框架题，重点强调热点内容、异步链路、回放污染和多机房稳定性。

### 9.1 热点内容与计数一致性
- 当评论、点赞、弹幕或互动量突然集中到少数热点内容时，系统最先会被哪里打爆？如何做热点隔离和削峰？
- 如果展示计数、缓存计数和最终落库结果短时间不一致，平台如何控制用户感知并在后续收敛？

### 9.2 异步审核与回放补偿
- 内容审核链路经常拆成多阶段异步处理，如果中间某一环卡住、重复消费或回放顺序错乱，平台如何保证状态最终正确？
- 当消息回放是修数手段也是风险来源时，怎样避免“补偿把旧脏数据再次放大”？

### 9.3 消息链路稳定性与送达率治理
- 消息系统既要保实时，又要保不丢不重，遇到高峰堆积时怎么做优先级、限流、降级和死信治理？
- 如果某个下游消费方长期处理慢、幂等差或失败重试不规范，平台侧如何做治理和责任边界收口？

### 9.4 多机房容灾与故障扩散控制
- 内容平台多机房部署时，哪些数据可以异步，哪些链路必须保证更强约束？
- 当单机房、单地域或核心依赖抖动时，如何防止评论、互动、消息链路出现跨机房级联故障？

### 9.5 工程治理与业务接入约束
- 如果评论、点赞、消息等不同业务团队接入同一套基础能力，但埋点、幂等键、重试策略、超时配置各不相同，平台如何推动统一治理？
- 平台团队怎样把“出事后排查困难”的链路改造成“平时可观测、回放可控、责任可追”的工程体系？

### 9.6 面试回答建议
- 回答内容平台后台研发题时，优先围绕：**热点隔离、异步状态机、回放治理、机房容灾、接入约束** 这 5 条主线组织答案。
- 不要只说“上 Redis / MQ 就行”，而要讲清楚：峰值怎么来、状态怎么漂、补偿怎么做、扩散怎么拦、治理怎么推进。

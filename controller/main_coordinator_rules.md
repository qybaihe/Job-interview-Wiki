# main_coordinator_rules.md

## Job-interview-Wiki 主线程协调规则

主线程不是播报员，而是协调器。

### 核心目标
- 保证 `Job-interview-Wiki` 可以**持续推进**
- 不允许只停留在目录初始化阶段
- 不允许只有计划文档，没有真实内容沉淀
- 不允许找到材料后长时间不落库

### 硬性推进要求
- **主线程每 20 分钟至少推进一次**
- 只要还有未完成抓取任务，就不能空转
- 每次介入至少完成以下动作之一：
  1. 新增一批抓取候选
  2. 落一批题库 Markdown
  3. 落一批面经 JSON
  4. 清洗 / 去重 / 归类已有材料
  5. 更新状态文件与计划文件

### 当前推进阶段
项目初期分成四类工作：
1. 线索发现（找面经来源）
2. 结构归类（公司 / 事业群 / 部门 / 岗位）
3. 内容落库（question-bank.md / interview-experiences/*.json）
4. 清洗去重（合并重复题、统一命名、补感悟）

### 触发介入条件（满足任一条立即介入）
- 20 分钟无新增 commit
- `controller/progress_state.json` 中 `running` 数量为 0
- `controller/progress_state.json` 中 `todo` 仍大于 0
- 有原始线索但尚未转成 Markdown / JSON
- 已落库内容存在明显重复或命名混乱
- 第一批优先抓取清单长期没有被消费

### 介入动作顺序
1. 先收残单（把已有线索整理进仓库）
2. 再优先处理第一批抓取清单中的高优先级公司/岗位
3. 再补满当前轮次的抓取 / 清洗任务
4. 子任务完成后立刻收口、commit/push
5. 更新 `controller/progress_state.json`
6. 若某条来源不稳定或噪音太高，记录后跳过，不阻塞整体推进

### 默认优先级
默认按如下顺序推进：
1. 头部公司（字节 / 腾讯 / 阿里 / 美团 / 拼多多）
2. 后端开发岗位
3. 能明确归到事业群 / 部门的真实面经原文
4. 能抽出 5 个以上有效问题的材料
5. 再扩到算法 / 客户端 / 测开 / 产品

### 内容落库规则
#### question-bank.md
适合沉淀：
- 高频基础题
- 算法题
- 项目题
- 系统设计题
- 反问方向

#### interview-experiences/*.json
适合沉淀：
- 某一次具体面试
- 题目数组
- 每题感悟
- 最终感悟
- 来源链接

### 异常处理
以下不算有效推进：
- 只新增目录，不新增内容
- 只复制原文，不清洗
- 只堆链接，不归类
- 只写大段散文，不写结构化 JSON
- 无法判断公司 / 岗位 / 事业群的低质量碎片

处理方式：
1. 不收口
2. 暂存为线索
3. 后续再二次清洗

### 主线程每轮必须判断
1. 本轮是否有新 commit？
2. 是否新增了可用题库内容？
3. 是否新增了结构化 JSON？
4. 第一批清单是否被持续消费？
5. 若没有推进：立即补位，不等待

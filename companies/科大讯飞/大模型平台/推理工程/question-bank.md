# 科大讯飞｜大模型平台｜推理工程｜question-bank

> 说明：本题库基于科大讯飞 / 星火官方公开文档做岗位能力映射整理。
> 当前更适合作为首个正式 question-bank 入口，不伪称为单场真实面试逐字题单。

## 岗位方向
- 大模型平台 / 推理工程
- 大模型在线服务 / 流式生成 / 模型网关
- 模型接入、协议、鉴权、路由、流控、容量治理

## 一、官方能力映射题（高相关）

1. 星火大模型的 WebSocket 调用为什么要单独做鉴权 URL 生成？它和普通 Bearer Token 鉴权相比有什么工程差异？
2. 科大讯飞官方 WebSocket 鉴权里，为什么要求 `date` 采用 RFC1123 格式，且时间偏差控制在 300 秒内？如果客户端时钟漂移会出现什么问题？
3. 鉴权串里为什么要把 `host`、`date`、`request-line` 一起参与 HMAC-SHA256 签名？如果只签 path 或只签时间，会有什么风险？
4. 如果你来实现讯飞星火的鉴权 SDK，如何避免不同语言实现时出现签名不一致的问题？
5. 官方文档里的 `authorization` 为什么要先拼接 origin，再做一次 base64 编码？这一步在网关侧如何校验？
6. 星火接口返回是流式的，`status=0/1/2` 分别代表什么？客户端为什么不能只处理最后一次返回？
7. 流式返回场景里，前端展示、服务端聚合、审计日志、计费统计应该分别在哪个阶段落点？
8. 如果流式输出过程中网络断开，客户端重连后应该从头重试还是断点续传？为什么？
9. 为什么官方文档强调“开发者需要将接口多次返回的结果进行拼接展示”？如果拼接逻辑写错，常见线上 bug 会是什么？
10. 星火文档里给出了 `question_tokens / prompt_tokens / completion_tokens / total_tokens`，这些指标分别适合支撑哪些运营或系统决策？
11. 官方文档要求控制历史上下文 token 长度。如果对话历史过长，你会怎么裁剪，才能兼顾成本、效果和时延？
12. 文档里提到 1 token 约等于 1.5 个中文汉字或 0.8 个英文单词。为什么工程上不能直接按字符数粗暴限流？
13. 讯飞给出了 Lite / Pro / Pro-128K / Max / Max-32K / Ultra / kjwx 等不同版本。你会怎样设计模型路由层，决定一个请求该走哪条链路？
14. `generalv3`、`generalv3.5`、`max-32k`、`pro-128k` 这些 domain 参数暴露了什么平台设计思路？为什么路由层不直接让用户传底层物理地址？
15. 如果某业务强依赖长上下文，但预算敏感，你会如何在 `Pro-128K`、`Max-32K`、普通版本之间做选型和降级？
16. 文档里存在“科技文献大模型（kjwx）”这样的垂直版本。平台层如何同时支持通用模型和垂直模型，而不把接入层做乱？
17. 讯飞官方错误码里有并发冲突、秒级流控、日流控、授权不足、服务忙、token 超长等。你会如何把这些错误分层：用户错误、配置错误、容量错误、平台内部错误？
18. 错误码 `10006/10007/11202/11203/10110/10907` 分别更适合用什么重试策略？哪些该立即失败，哪些该指数退避？
19. 文档提示“同一用户不能多处同时连接”。为什么推理服务会限制同用户并发连接？这背后可能涉及哪些状态管理问题？
20. 为什么大模型在线服务的限流不能只看 QPS，还要看 token、上下文长度、生成长度和流式连接占用时长？

## 二、推理平台工程化（高频追问）

21. 科大讯飞这类大模型平台如果同时服务多个业务线，网关层如何做租户隔离、鉴权、配额和审计？
22. 在线推理和批量推理的核心差异是什么？如果一个平台两者都支持，资源池应该怎么隔离？
23. 流式输出场景里，你更关注 TTFT、TPOT、总完成时长，还是连接保持时长？分别对应什么问题？
24. 如果用户感知“回复开始得很快，但后半段明显卡顿”，你会优先怀疑模型解码、调度队列、网络 flush，还是上游消费逻辑？
25. 一个推理平台为什么必须把“鉴权成功率、首 token 延迟、每秒输出 token 数、错误码分布、token 计费”一起纳入观测？
26. 如果某版本模型刚上线后 `10110 服务忙` 暴涨，你会如何判断是容量不足、路由异常、灰度策略失衡还是流量突刺？
27. 如果多个模型版本共用一套网关，怎么设计灰度发布和快速回滚，避免新版本影响全局可用性？
28. 当某些请求上下文极长、生成也很长时，为什么它们会拖坏整个平台的尾延迟？应该如何隔离？
29. 线上为什么要区分“首包快但总时长长”和“首包就慢”两类问题？它们通常落在哪些不同瓶颈点？
30. 多租户推理平台做容量评估时，为什么不能只看日均流量，还必须看峰值并发、token 分布、长短请求占比和模型版本结构？

## 三、系统设计题

31. 设计一个面向外部开发者的大模型接入网关，支持 WebSocket / Web API、鉴权、配额、流式输出和审计。
32. 设计一套星火类大模型的客户端 SDK，要求支持鉴权生成、重试、流式拼接、错误处理和多模型版本切换。
33. 设计一个多模型路由系统，支持 Lite / Pro / Max / 长上下文版本 / 垂类模型的自动选择与降级。
34. 设计一套大模型 token 计量与计费系统，如何保证 `prompt_tokens / completion_tokens / total_tokens` 的统计准确性？
35. 设计一套线上限流与背压系统，既能保护 GPU/引擎容量，又能给上游返回可解释的错误信息。
36. 设计一个推理平台的可观测系统，要求覆盖流式链路、错误码、鉴权失败、模型版本、token 消耗和租户维度分析。

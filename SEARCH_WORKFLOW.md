# SEARCH_WORKFLOW.md

## 目标
把 `Job-interview-Wiki` 的信息收集工作固定成一条可以持续执行的生产路径，而不是临时搜索、临时整理。

本项目最终只沉淀两类结果：
1. `question-bank.md`
2. `interview-experiences/*.json`

所以所有查找动作都必须先回答一个问题：
> 当前这份材料最终是进题库，还是进结构化面经 JSON？

---

## 一、查找顺序（固定路径）

每轮查找都按以下顺序进行，不能乱跳：

### 第一层：完整面经长帖（最高优先级）
优先寻找：
- 明确公司名
- 明确岗位
- 最好明确事业群 / 部门
- 明确一面 / 二面 / 三面流程
- 能抽出 5 个以上有效问题

典型用途：
- 直接生成 `interview-experiences/*.json`
- 再从 JSON 中反向抽题补 `question-bank.md`

---

### 第二层：GitHub / Markdown 整理仓库
优先寻找：
- 大厂某岗位面试题整理
- 多篇面经交叉整理稿
- 高频题总结帖
- interview / 面经 / 后端 / 算法类仓库

典型用途：
- 生成 `question-bank.md`
- 补充高频基础题、项目题、系统设计题

---

### 第三层：个人博客 / 长文复盘
优先寻找：
- 一整场面试复盘
- 明确项目追问
- 面试后反思 / 感悟
- 系统设计题展开

典型用途：
- 补 JSON 里的 `insight`
- 补题库中的项目题 / 系统设计题 / 反问方向

---

### 第四层：碎片来源（低优先级）
包括：
- 评论区补充
- 零散短帖
- 单题回忆
- 无完整上下文的问答

典型用途：
- 只能作为题库补充线索
- 不直接生成 JSON

## 四、查到后如何处理

### 路径 A：完整长帖
判定标准：
- 公司明确
- 岗位明确
- 有轮次或题目顺序
- 能抽出多道题

处理方式：
1. 写入 `interview-experiences/*.json`
2. 提炼高频题补 `question-bank.md`
3. 保留来源链接

### 路径 B：多篇题库整理
判定标准：
- 没有单场完整流程
- 但题目稳定、重复出现
- 可归纳为共性问题

处理方式：
1. 去重
2. 合并同义题
3. 写入 `question-bank.md`

### 路径 C：碎片线索
判定标准：
- 只有单题
- 只有评论区零散回忆
- 岗位/事业群不清晰

处理方式：
1. 暂存为线索
2. 不直接生成 JSON
3. 等后续补来源后再决定是否入题库

## 五、每轮 cron 的消费规则

### 主 cron：jobwiki-orchestrate
建议频率：每 20 分钟

每轮必须执行：
1. 读取 `controller/progress_state.json`
2. 判断还有哪些优先目标没有真实题库 / JSON
3. 只选择 1~2 个目标推进，避免发散
4. 按“完整面经 → GitHub整理 → 博客复盘 → 碎片补充”的顺序查找
5. 本轮至少形成一个真实产物：
   - 1 个 `question-bank.md`
   - 或 1 份 `interview-experiences/*.json`
   - 或 1 批去重清洗后的问题补充
6. 更新状态文件
7. 输出下一轮推荐目标

### 辅 cron：jobwiki-report
建议频率：每 1~2 小时

职责：
- 汇总最近提交
- 汇总新增题库数量
- 汇总新增 JSON 数量
- 汇总还卡在哪些公司 / 岗位

只做简报，不负责生产。

## 七、不可违反的规则

1. 不要把整篇原文直接塞进 JSON。
2. 不要把碎片回忆当完整面经收录。
3. 不要只堆链接，不做清洗。
4. 不要把同义问题重复写进题库。
5. 不要在没有来源的情况下写强断言。
6. 每轮必须有真实产出，不能只播报状态。

---

## 八、主线程介入原则

当出现以下情况，主线程必须介入补位：
- 20 分钟没有新增真实内容
- `running = 0` 且 `todo > 0`
- 优先公司长期未被消费
- 只产出计划、没有产出内容
- 只有摘要，没有 commit

主线程介入后优先顺序：
1. 收残单
2. 处理当前第一优先公司
3. 先补题库或 JSON 之一
4. 立即 commit / push
5. 更新 `progress_state.json`

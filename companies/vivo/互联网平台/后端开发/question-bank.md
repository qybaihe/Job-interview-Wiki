# vivo / 互联网平台 / 后端开发 - Question Bank

## 岗位方向
- vivo 互联网平台后端通常会落在：账号中心、消息中心、活动平台、会员权益、内容/服务分发、运营中台等场景。
- 相比纯交易型后端，更强调：设备生态、多端登录、系统版本兼容、端云协同、峰值活动治理、国内/海外多区域治理。
- 面试追问往往不只停留在 Java / MySQL / Redis 八股，还会看你是否理解手机厂商平台型后端的业务形态。

## 基础能力
1. 手机厂商的“互联网平台后端”通常会服务哪些业务域？为什么它和纯交易型后端不完全一样？
2. 如果一个平台同时支撑账号、消息、活动、会员、内容分发，服务边界应该如何划分？
3. 终端厂商的互联网平台为什么通常既要看 DAU，也要看设备活跃、系统版本和地区分布？
4. 多终端、多版本并存时，后端接口演进应如何做兼容设计？
5. 你会如何划分“通用平台能力”和“具体业务线能力”？
6. 如果平台提供统一登录、消息中心、活动配置、权益中心，最适合沉淀成平台层的公共能力有哪些？

## 账号、设备与安全
7. 设计一个统一账号中心，支持手机号、验证码、设备信任、多端登录和风控校验。
8. 账号体系、设备标识、登录态三者在手机厂商平台里如何配合？
9. 如果同一用户在多台设备上频繁切换登录，后端如何做安全判断而不过度伤害体验？
10. 如果出现批量盗号、验证码轰炸或撞库，平台后端应该如何快速止损？
11. 账号风控误伤正常用户时，申诉、恢复和审计链路应该怎么设计？
12. 在“设备可信”和“账号可信”不完全一致时，风控策略应该如何分层？

## 消息、活动与会员平台
13. 设计一个站内消息 / Push 平台，需要支持运营消息、系统消息、行为触发消息和优先级控制。
14. 如果推送到达率突然下降，你会优先检查哪些链路节点？
15. 大促或新品发布期，活动平台最容易出现哪些稳定性问题？
16. 会员权益中心如何处理权益发放、幂等、过期、回收和补偿？
17. 如果活动玩法很多、配置项复杂，如何设计一个低风险的运营配置平台？
18. 为什么运营活动系统往往更依赖灰度、开关、限流和回滚能力？
19. 如果活动配置已经发布，但只在某些 app 版本或机型上异常，平台后端应如何定位？

## 内容分发与端云协同
20. 如果平台里既有资讯/内容流，也有会员/权益/活动入口，推荐和分发策略要考虑哪些目标冲突？
21. 手机厂商平台的内容分发为什么不能只看点击率？还要看哪些长期指标？
22. 终端版本升级后，如果客户端请求参数变化，后端如何兼容旧版本并快速验证新版本效果？
23. 如果终端问题无法稳定复现，服务端如何通过日志、trace、配置快照帮助定位？
24. 为什么手机厂商平台后端常被问到“端云协同”？你会怎么解释这类系统的难点？
25. 对手机厂商的互联网平台来说，什么叫“接口兼容”，什么叫“行为兼容”？

## 全球化与多区域治理
26. 如果 vivo 的互联网平台同时面向国内与海外地区，后端如何做多区域部署与数据隔离？
27. 海外网络质量波动较大时，接口超时、重试、回执和降级要怎么设计？
28. 哪些用户数据适合跨地域同步，哪些数据必须本地落地？为什么？
29. 对账号、消息、会员这类公共平台，合规和容灾设计分别要优先解决什么问题？
30. 如果某个区域网络异常导致消息大量堆积，平台后端如何做削峰、优先级调度和补偿？

## 项目深挖与追问链
31. 你做过最接近“平台中台 / 公共能力平台”的项目是什么？在其中承担了什么职责？
32. 如果你过去做的是普通互联网后端，如何把经历迁移表达为适合 vivo 互联网平台的能力？
33. 你认为手机厂商互联网平台的后端难点，更偏高并发、复杂业务编排、端云协同还是全球化治理？为什么？
34. 如果面试官追问“为什么想做 vivo 这类公司的平台后端”，你会如何结合业务形态回答？
35. 对 vivo / OriginOS 生态相关平台岗位，为什么面试里既会问分布式基础题，也会问业务平台治理题？

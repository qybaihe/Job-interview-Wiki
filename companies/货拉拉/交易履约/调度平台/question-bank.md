# 货拉拉｜交易履约 / 调度平台｜question-bank

> 说明：本题库基于货拉拉 / Lalamove 官方公开业务页面与开放平台能力整理，适合作为“交易履约 / 调度平台”方向的正式 question-bank。
> 证据边界：
> - 主要来自官方业务介绍、Business 页面、API Solutions 页面；
> - 更偏“公开业务映射型题库”，不伪称为单篇真实逐轮面经。

## 岗位方向
- 交易履约
- 调度平台
- 运力分配
- 开放平台 / 企业物流集成
- 后端开发 / 服务端研发

## 一、交易与履约状态机

1. 对于货拉拉这种 on-demand delivery platform，为什么“下单成功”和“履约成功”必须拆成两个阶段？
2. 如果订单需要经历待接单、已接单、司机到达、装货中、运输中、卸货中、已完成、异常取消等状态，状态机该如何设计？
3. 当司机接单后又取消，订单状态、调度任务和补偿策略分别应该怎样回滚？
4. 如果用户在未接单、已接单待到达、装货中、运输中四个阶段申请取消，系统应如何差异化处理？
5. 为什么即时货运的履约状态机会比普通电商交易状态机复杂得多？
6. 在多点配送场景下，为什么订单状态机通常不能只保留“待接单 / 配送中 / 已完成”三态？
7. 如果一个多点配送订单中途某个站点取消，剩余站点如何继续履约？账务和运费如何重算？
8. 多点订单里，如果司机已经完成前两个站点，第三个站点临时地址变更，系统如何保证状态、路线和报价同步更新？

## 二、调度与运力分配

9. 官方强调平台连接 people / vehicles / freight / roads。后端领域模型里你会如何抽象用户、司机、车辆、订单、运单、路线、报价之间的关系？
10. 平台同时服务 individuals、small businesses 和 corporations。C 端与 B 端在订单建模、权限体系、结算链路上的核心差异是什么？
11. 官方提到 wide fleet of delivery vehicles。你会怎样设计车型、载重、容积、禁限行、服务范围等多维属性模型？
12. 多车型即时调度场景下，调度系统如何在“更快接单”“更低成本”“更高履约成功率”之间平衡？
13. 如果同一订单可由面包车或小货车承接，推荐引擎如何在报价与履约成功率间做取舍？
14. 当一个城市运力紧张时，调度系统如何区分高优先级订单、普通订单与可延期订单？
15. 多点配送的路径规划为什么比单点送货复杂得多？你会如何做 stop 排序与局部重排？
16. 多点配送订单的 ETA 应该如何计算？是给整单 ETA、每站 ETA，还是二者都要？
17. 如果司机位置上报频率太高，系统如何做采样、压缩或冷热分层？
18. 实时调度场景里为什么不能把所有决策都放在数据库事务里完成？

## 三、开放平台与企业物流集成

19. API 支持 backend-to-backend seamless order placements。如果商家系统直接从后端下单到货拉拉平台，开放平台鉴权如何设计？
20. API 同时支持 point-to-point 与 multi-stop orders，接口层如何兼容两类订单而不让字段语义混乱？
21. API 支持 schedule pick up time up to 30 days in advance。预约单和即时单在运力预占、状态机、调度策略上有哪些差异？
22. API 支持 instant quotation of delivery fees。报价接口应该如何设计才能既快又可追溯？
23. 报价受车型、距离、时段、站点数、附加费影响时，计价引擎如何做规则配置与灰度发布？
24. API 支持 order cancellation。取消接口怎样设计幂等，才能避免重复取消和状态脏写？
25. API 通过 Webhook 回推 delivery status updates。Webhook 体系如何保证消息可靠投递、幂等消费与重试？
26. 如果商家回调地址超时或返回 500，Webhook 平台如何做退避重试和死信管理？
27. Webhook 事件乱序到达时，订单平台如何避免状态回滚？
28. 如果企业客户要接 ERP / WMS / OMS，开放平台为什么比普通 App 下单更强调审计、限流、签名和回放保护？
29. API 订单和 App 订单并存时，统一订单中心如何避免多入口造成的状态割裂？
30. Partner Portal 与 API Key 体系如何设计权限分层，避免商家误用生产密钥？

## 四、系统设计与稳定性

31. 为什么即时货运平台通常要拆成订单中心、调度中心、计价中心、司机中心、定位中心、开放平台几个核心域？
32. 司机抢单 / 系统派单并存时，如何避免同一司机被多单并发锁定？
33. 如果一个司机刚被系统匹配成功，但客户端网络延迟导致确认慢，系统如何处理超时释放与二次分配？
34. 预约单在未来 30 天内可能反复改期，系统如何设计任务编排与重算机制？
35. 实时司机位置、Webhook 事件、报价结果、订单状态，这四类数据在存储形态上为什么不应该完全一样？
36. 当暴雨、节假日、搬家旺季导致供需失衡时，货运平台最重要的降级策略有哪些？
37. 如果开放平台客户批量重试下单，导致重复创建订单，系统如何通过业务幂等键止损？
38. 多点配送与预约单叠加后，为什么测试复杂度会指数上升？你会怎样设计测试矩阵？
39. 企业客户要求月结、账期、对账与发票能力时，账务系统如何从 C 端即时支付模型演化出来？
40. 设计一个支持开放平台接入、即时/预约、多点配送、Webhook 回调和实时司机位置的货运平台后端架构。

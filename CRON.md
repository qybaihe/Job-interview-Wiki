# CRON.md

## Job-interview-Wiki 持续推进设计

目标：
- 让项目可以一直推进下去
- 不依赖一次性人工指令
- 主线程收到摘要后，能知道这一轮该干什么

---

## 推荐 cron 拆分

### 1. jobwiki-orchestrate（主 cron）
建议频率：**每 20 分钟一次**

执行入口：
- `python3 scripts/jobwiki_orchestrate.py`

职责：
1. 读取 `controller/progress_state.json`
2. 扫描首批重点目标是否已有真实 `question-bank.md`
3. 扫描是否已有真实 `interview-experiences/*.json`
4. 输出本轮应推进的目标（公司 / 岗位 / 事业群 / 动作类型）
5. 主线程据此决定：
   - 找线索
   - 整题库
   - 落 JSON
   - 清洗去重

这个 cron 的核心不是“汇报”，而是：
> **生成下一轮明确动作，并驱动主线程继续推进**

---

### 2. jobwiki-report（辅 cron）
建议频率：**每 1~2 小时一次**

职责：
- 汇总最近提交
- 汇总新增了哪些公司 / 题库 / JSON
- 汇总当前还卡在哪些公司或岗位

它只做简报，不负责生产推进。

---

## 主线程每轮标准动作

收到 orchestrate 摘要后，主线程必须做以下判断：

1. 是否已有新增线索待整理？
2. 是否应优先给某家公司补 question-bank.md？
3. 是否已有足够信息可以生成一份 interview-experiences JSON？
4. 是否存在重复内容需要清洗？
5. 若本轮没有真实产出，必须主动补位，不能只发摘要。

---

## 推荐推进节奏

### 第 1 阶段
先做：
- 字节 / 腾讯 / 阿里 / 美团 / 拼多多
- 后端开发

### 第 2 阶段
再补：
- 算法
- 客户端
- 测开
- 产品

### 第 3 阶段
再扩公司：
- 快手
- 京东
- 百度
- 以及后续更多中大厂

---

## 成功标准

一个“有效推进轮次”至少满足以下之一：
- 新增 1 个题库文件的有效内容
- 新增 1 份结构化面经 JSON
- 完成 1 批线索的清洗归类
- 明确推进了某家公司 / 事业群 / 部门的真实内容

如果一轮 cron 之后什么都没新增，只是重复状态，那就不算有效推进。

# T3出行｜出行平台 / 后端开发｜question-bank

> 状态：verified-question-bank
> 适用岗位：T3出行 / 出行平台 / 后端开发 / Java后端 / 服务端研发
> 证据边界：本稿基于可访问的牛客搜索页可复核线索整理，已确认搜索结果中存在 **“t3出行 / t3出行面试 / t3出行java / t3出行实习 / t3出行数据分析 / t3出行算法”** 等关联项。当前版本作为 **首版正式 question-bank** 收录，定位为基于公开线索做的保守归纳，不冒充逐字逐句原题回放。

## 一、出行平台业务高信号题

1. 网约车平台的订单状态机一般怎么设计？从下单、派单、司机接单、到达、开始计费、结束行程到支付完成，哪些状态必须严格单向流转？
2. 如果同一时间司机取消、乘客取消、系统超时取消三类事件并发到来，后端怎么保证订单最终状态唯一？
3. 派单系统里为什么不能只靠数据库锁来控制抢单？什么时候会引入缓存、消息队列或本地限流？
4. 司机接单场景如何防止“同一司机接到两单”或者“同一订单被多个司机接走”？
5. 如果用户端显示“司机已接单”，但司机端其实没看到新单，可能是哪几层出了问题？你如何排查？
6. 为什么出行平台特别强调幂等？请结合下单、支付回调、优惠券发放、退款、消息消费几个场景说明。
7. 实时位置上报为什么通常不会直接落 MySQL 主链路？更常见的存储和查询策略是什么？
8. 如何设计“附近司机查询”？如果司机位置持续变化，怎样兼顾查询效率和数据新鲜度？
9. 网约车价格预估为什么常常拆成计价、活动、券、动态加价几个模块，而不是一个大接口做完？
10. 如果高峰期某城市瞬时下单暴涨，平台后端首先会保护哪些核心链路，牺牲哪些非核心能力？
11. 出行平台为什么比普通电商更依赖实时性？这种实时性会怎样影响缓存、超时、重试和监控设计？
12. 如果乘客投诉“平台显示行程完成，但实际没上车”，你会从哪些日志和事件流里追溯？

## 二、订单 / 派单 / 履约场景深挖

13. 派单系统里“广播抢单”和“集中调度派单”各适合什么业务阶段？
14. 如果司机接单后长时间未到达，系统里的超时关闭、重新派单、乘客补偿应该怎么协同？
15. 订单履约过程中，为什么要区分主状态、展示状态、结算状态？三者混在一起会导致什么问题？
16. 当旧事件晚到时，比如“司机到达”在“订单取消”之后才消费到，系统如何让旧事件失效？
17. 如果派单结果通过 MQ 异步通知多个系统，怎样避免部分系统成功、部分系统失败造成状态分裂？
18. 出行平台里的延迟任务通常有哪些：接单超时、支付超时、司机到达超时、行程异常关单、退款检查？这些任务更适合怎么实现？
19. 订单链路里 TraceId、订单号、司机ID、城市ID、渠道ID 为什么都应该是核心排障字段？
20. 如果一个城市因为第三方地图服务抖动导致 ETA 全部异常，你会怎样快速止损？

## 三、支付 / 结算 / 对账 / 风控

21. 出行平台的支付链路为什么要特别关注“重复支付回调”和“支付成功但订单未完成”这两类问题？
22. 司机分成、平台抽佣、补贴、优惠券抵扣、退款等金额计算为什么通常不建议散落在多个服务里？
23. 对账系统中，订单表、支付表、退款表、账单表、第三方支付流水表之间怎么建立校验关系？
24. 如果 T+1 对账发现差异单，你会如何区分是延迟到账、重复记账、金额口径错误，还是状态流转错误？
25. 退款为什么不能简单做成“收到请求就直接退”？需要先判断哪些订单与支付状态？
26. 风控系统在网约车场景里会重点看哪些信号：刷单、虚假行程、设备异常、频繁取消、位置漂移、账号关联？
27. 如果优惠券和支付并发，如何避免“优惠券已核销但订单未支付成功”的资金风险？
28. 对账补偿任务为什么必须有重试上限、人工兜底和审计日志？

## 八、面试回答建议：T3出行方向应该打出的主线

1. **优先强调业务理解**：不是只背八股，而是要把订单、派单、司机状态、位置、支付、结算串成完整链路。  
2. **突出一致性与并发控制**：幂等、防重、旧事件失效、缓存一致性、MQ 可靠性，是出行平台后端的高频核心。  
3. **讲清实时性与高峰保护**：出行平台的价值在于实时响应，答题时要体现限流、降级、超时、隔离、监控。  
4. **把系统设计题讲成业务闭环**：不要只说技术组件，要说明为什么这样设计能支撑城市级高峰和复杂履约。

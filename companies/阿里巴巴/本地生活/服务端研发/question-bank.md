# 阿里巴巴 / 本地生活 / 服务端研发 / 题库

> 说明：当前未检索到足够高质量、可直接引用的一手公开面经正文，因此本题库作为保守型补位候选，基于公开可验证业务背景（到店 / 到家 / 交易履约 / 商家平台 / 营销活动）与阿里系相近岗位题库风格整理；后续仍应继续补充真实面经来源。

## 基础知识
- Java 中 `synchronized`、`ReentrantLock`、AQS 的实现原理分别是什么？
- JVM 内存模型（JMM）解决了什么问题？`volatile` 的可见性与有序性如何体现？
- MySQL InnoDB 的聚簇索引和二级索引有什么区别？为什么回表会影响性能？
- MySQL 事务隔离级别分别会带来哪些问题？RR 是如何尽量避免幻读的？
- Redis 在高并发读写场景中常见问题有哪些？如何处理热 key、大 key、缓存击穿与雪崩？
- MQ 在订单、履约、营销链路中的典型作用是什么？如何保证消息不丢、不重、不错序？
- 分布式锁常见实现方式有哪些？Redis 分布式锁会有哪些风险？
- 接口幂等如何设计？下单、支付、退款等场景分别要注意什么？
- 限流、熔断、降级、隔离分别解决什么问题？
- 线上 Full GC 频繁时，你会如何定位是内存泄漏、对象激增还是参数配置问题？

## 业务链路
- 设计一个外卖 / 到店订单状态机，需要考虑哪些状态、逆向流转和补偿逻辑？
- 履约链路中“下单成功但后续库存 / 骑手 / 商家确认失败”时，系统如何兜底？
- 优惠券、红包、满减、会员折扣叠加时，价格计算服务怎么设计才能既灵活又可审计？
- 商家侧商品、门店、营业时间、配送范围频繁变更，如何保证前台展示与交易校验一致？
- 如果一个城市在饭点流量暴涨，如何做就近限流、容量预估和资源扩容？
- 本地生活场景为什么比普通电商更强调“时效性”和“空间属性”？这会如何影响服务端设计？

- 设计一个高并发下单系统，要求防重、幂等、可恢复。
- 设计一个优惠券发放与核销系统，要求支持高并发抢券、防刷和审计。
- 设计一个门店商品聚合服务，要求低延迟、强可用，并能承受热点门店流量。

- 设计一个营销活动平台，要求支持多活动并发、规则配置和灰度投放。

# 海尔 / 新事业生态圈 / 后端开发 question bank

> 说明：本题库基于可见公开岗位线索与公开 Java 后端面经整理，定位为 **首版 question-bank**。
> 当前更适合作为新增公司破冰入口；后续若补到更多可稳定核验的公开长帖，再继续补 `interview-experiences/*.json`。

## 一、项目与业务理解

1. 介绍一下你的项目，说一下项目的整体框架。
2. 你的项目有没有真正落地？如果有，实际使用场景是什么？
3. 如果项目没有落地，你如何证明它不是只停留在简历上的 demo？
4. 在实习中你负责什么，具体做了哪些工作？
5. 在项目推进过程中，你遇到过什么技术难点，最后怎么解决？
6. 如果让你做海尔这类制造业 / 智慧家庭企业的后端系统，你会怎么平衡“业务可落地”和“平台可复用”？

## 二、后端基础与微服务

7. 介绍一下微服务，为什么要做微服务拆分？
8. 微服务拆分之后，接口治理、服务发现、配置管理、链路追踪通常怎么做？
9. 你项目中用到的集合类主要是哪些？为什么这样选？
10. 说一下 List 和 Map 的区别，以及各自适合什么使用场景。
11. 如果线上接口延迟升高，你会优先排查集合使用、锁竞争还是数据库瓶颈？为什么？
12. 如果某个服务在流量高峰期频繁超时，你会如何做限流、降级和隔离？

## 三、缓存、Redis 与数据增长

13. 为什么选择用 Redis？它在你的项目里承担了什么角色？
14. Redis 和数据库分别适合放哪类数据？
15. Redis 继续深问时，你会如何回答数据结构、持久化、淘汰策略、缓存一致性、热点 key、穿透 / 击穿 / 雪崩？
16. 如果后续数据量突然变大，你的系统怎么处理？
17. 当数据库压力持续上升时，你会如何做后续数据库优化？
18. 如果缓存失效、数据库被打满，系统如何限流、降级与恢复？
19. 如果是制造业设备数据或家电状态频繁写入，缓存和数据库该如何分工？

## 四、数据库与网络基础

20. 如果数据规模继续上涨，数据库先从索引、SQL、分库分表、读写分离中的哪一层开始优化？
21. 你的项目里有没有做过数据库优化？具体做法和收益是什么？
22. 说一下 TCP 三次握手。
23. 如果把三次握手放到真实业务排障场景里，你会如何解释连接建立慢、半连接队列、重传等现象？
24. 线上慢 SQL 的常见成因有哪些？你会如何借助执行计划定位问题？
25. 如果事务链路跨多个服务，怎样在不引入重型分布式事务的情况下保证最终一致性？

## 五、智慧家庭 / IoT 追问

26. 如果后续落到智慧家庭 / 物联网平台，你会如何设计设备数据上报链路？
27. IoT 场景下如何做高并发消息接入、设备鉴权与状态同步？
28. 家电联网失败或设备状态延迟同步时，你会怎么做端到端排障？
29. 如果同一个家庭里有空调、冰箱、洗衣机等多类设备，统一设备模型应该怎么抽象？
30. 设备离线、弱网、重复上报、状态乱序这些问题，后端通常怎么处理？
31. 如果要做“回家自动开空调”这类联动规则，你会如何设计规则引擎和执行链路？

## 六、行为与岗位匹配

32. 你和科班同学相比，差距怎么弥补？你的优势是什么？
33. 为什么选择海尔 / 制造业 / 智慧家庭相关方向，而不是互联网纯流量业务？
34. 如果让你做家电 / 智慧家庭 / 生态协同类后端，你觉得和传统互联网后端最大的不同是什么？
35. 你更想做偏业务系统、平台中台，还是偏设备云 / IoT 基础设施？为什么？
36. 入职后如果先接一个你不熟悉的设备侧业务，你会如何快速补齐领域知识并推进交付？

## 七、系统设计题

37. 设计一个智慧家庭设备接入平台，支持设备注册、鉴权、状态上报和指令下发。
38. 设计一个家庭场景联动系统，支持定时触发、条件触发、失败重试和执行日志审计。
39. 设计一个售后 / 设备故障诊断后端系统，要求支持告警聚合、工单流转和状态同步。
40. 设计一个制造业企业的统一账号与权限体系，兼顾 C 端用户、B 端渠道与内部运营人员。

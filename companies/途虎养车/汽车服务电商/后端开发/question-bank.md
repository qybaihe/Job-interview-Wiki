# 途虎养车｜汽车服务电商｜后端开发｜question-bank 候选

> 说明：本候选仅基于公开可访问来源整理，采用“公司业务公开信息 + 公开面经标题/摘要片段”的方式构建题库，**不冒充逐字还原真实面试原文**。
>
> 本轮选择途虎养车，原因：在仓库现状中它仍为未覆盖公司，而马蜂窝、汽车之家已存在 question-bank；同时公开来源足以支持构建“汽车服务电商 / O2O 履约 / 门店网络 / 供应链”语境下的后端题库。

## A. 业务理解 / 平台抽象

1. 途虎养车是一个“线上预约 + 线下安装/服务”的 O2O 汽车服务平台。如果让你抽象它的核心后端领域模型，你会如何拆分商品、订单、库存、履约、门店、技师、用户、营销这几个域？
   - 来源映射：公开资料将途虎描述为 O2O 汽车服务平台，主营轮胎、机油、保养、维修、美容等服务。

2. 相比传统纯电商，途虎的订单不是“发货即结束”，而是要和门店服务预约、到店核销、施工交付联动。你会如何设计订单状态机？
   - 来源映射：公开资料明确强调线上下单、线下享受服务的一体化模式。

3. 途虎早期从轮胎切入，后续扩展到 30 余种汽车服务。如果业务从单品扩展到多服务品类，后台商品与服务能力该如何抽象，避免模型越做越脏？
   - 来源映射：公开资料提到最初从轮胎业务切入，2014 年扩展到 30 余种汽车服务。

4. 如果一个订单同时包含轮胎、机油和门店服务工时，后端该如何拆单、计价和履约协同？
   - 来源映射：汽车后市场业务同时包含实物配件与线下服务，属于强场景化履约问题。

## B. 门店网络 / 履约 / 调度

5. 截至公开资料时间点，途虎拥有大量工场店和合作门店。若你负责“下单后门店分配”服务，会综合哪些因素做门店推荐？
   - 来源映射：公开资料提到 4,770 家工场店与 19,624 家合作门店。

6. 同一个城市里，用户下单后可能面对“最近门店没库存、但隔壁仓有货；另一个门店有档期但距离更远”的情况。你会如何做库存与预约的一体化调度？
   - 来源映射：O2O 汽车服务天然涉及库存、配送与门店预约联动。

7. 对途虎这类平台来说，“门店推荐”为什么不能只按距离排序？如果要兼顾转化率、履约成功率和用户体验，你会怎么设计排序策略？
   - 来源映射：多门店网络下的服务质量、库存可得性、排班能力都会影响最终体验。

8. 如果用户预约 10:00 到店，但商品配送晚到、技师也临时排满，后端系统如何做改约、赔付、通知与 SLA 追踪？
   - 来源映射：汽车服务履约链路长，异常处理是核心后端场景。

9. 途虎早期自建过配送队伍来保证轮胎配送时效。若今天你来设计同城履约链路，会如何划分中心仓、前置仓、干线、末端配送与门店收货责任？
   - 来源映射：公开资料提到 2012 年在上海设立物流中心并自建配送队伍。

## C. 供应链 / 库存 / 价格

10. 轮胎、机油等商品既有标准化 SKU，也存在车型适配问题。你会如何设计商品中心与适配关系，保证“可卖”与“可装”一致？
   - 来源映射：途虎主营轮胎、机油、保养等强适配型汽车后市场商品。

11. 如果热门轮胎在大促期间被多个城市门店同时抢占库存，你会如何避免超卖？
   - 来源映射：汽车配件库存跨仓跨门店分配，超卖风险高。

12. 途虎还与腾讯等在智慧零售、供应链金融方面有公开合作线索。如果订单链路要接入赊销/账期/结算能力，后台系统边界该怎么定？
   - 来源映射：公开资料提到与腾讯在智慧零售、供应链金融方面合作。

13. 一个“价格中心”如何支持总部统一活动价、区域价、门店补贴价和会员价共存？
   - 来源映射：全国性平台 + 门店网络意味着价格体系复杂。

14. 如果运营希望对“同城、同车型、同服务包”实时比价并动态调价，后端该如何防止价格缓存脏读和门店侧价格认知不一致？
   - 来源映射：汽车服务电商中价格与履约门店展示强绑定，属于典型后端治理题。

## D. 高并发 / 稳定性 / 交易链路

15. 公开面经摘要里出现了“首次进入网站特别慢，第二次访问较快，而且不同地区用户体验不同”的场景。你会从 CDN、DNS、静态资源、缓存、冷启动等哪些角度排查？
   - 来源映射：牛客公开搜索结果片段出现该场景题。

16. 如果平台在双 11、618、换季保养节点流量暴涨，订单服务、库存服务、预约服务中你认为谁最容易先出问题？为什么？
   - 来源映射：汽车后市场业务有明显促销和季节性高峰。

17. 面经里多次出现秒杀、分布式锁相关问题。若“爆款轮胎限量秒杀 + 门店安装名额有限”同时存在，后端怎么做一体化限流与防重？
   - 来源映射：公开面经摘要出现“秒杀场景下加锁逻辑、本地锁/分布式锁”等问题。

18. 如何设计一个幂等的下单接口，避免用户重复支付、门店重复接单、库存重复扣减？
   - 来源映射：O2O 交易链路核心高频题。

19. 如果履约链路使用 MQ 解耦：下单、扣库存、预约门店、生成配送任务、发送通知分别适合如何拆成事件？怎样保证消息可靠性？
   - 来源映射：公开面经摘要出现 RocketMQ 可靠性、重复消费、消息丢失、消息堆积等问题。

20. 当某城市门店系统故障导致大量预约失败时，平台如何做熔断、切流、人工兜底和用户补偿？
   - 来源映射：多门店平台对区域性故障隔离要求高。

## E. Java / 中间件 / 基础能力

21. 公开面经里多次出现 HashMap、ThreadLocal、synchronized、ReentrantLock。你会怎样比较这些技术点在真实订单/预约系统中的使用边界？
   - 来源映射：牛客公开结果中多条面经摘要包含 HashMap、ThreadLocal、锁相关问题。

22. 为什么说 ThreadLocal 在请求上下文传递里“方便但危险”？在异步线程池和链路追踪场景中有哪些坑？
   - 来源映射：公开面经摘要点名 ThreadLocal 原理与使用。

23. 如果订单查询 SQL 在亿级数据量下变慢，针对 `user_id + status + create_time` 这类查询模式，你会如何设计索引并解释执行路径？
   - 来源映射：公开面经摘要出现亿级 orders 表 SQL 优化、联合索引顺序、回表等问题。

24. MySQL 做分库分表后，用户希望看“最近 10 条已支付订单”，你会如何支持查询、排序与分页？
   - 来源映射：公开面经摘要出现分表、索引、查询优化问题。

25. Redis 在途虎这种平台里适合缓存哪些数据，不适合缓存哪些数据？
   - 来源映射：公开面经摘要出现 Redis 为什么快、缓存穿透/雪崩等问题；平台场景包含门店页、商品页、适配结果、预约可用档期等热点读。

26. 如果库存服务出现缓存穿透、缓存击穿、缓存雪崩，你会分别怎么处理？
   - 来源映射：公开面经摘要明确出现缓存穿透、缓存雪崩与布隆过滤器问题。

27. 面经里频繁出现线程池、JVM、GC。若预约峰值期间接口 RT 抖动，你会怎样从线程池配置、GC、锁竞争、IO 多路复用几方面定位问题？
   - 来源映射：公开面经摘要包含线程池核心参数、JVM、GC 排查、IO 多路复用、select/poll/epoll 等。

## F. 数据 / 推荐 / 门店智能化

28. 平台拥有海量注册用户、交易用户和门店网络数据。如果要做“最适合你的保养套餐”推荐，后端需要沉淀哪些画像与特征？
   - 来源映射：公开资料披露注册用户、交易用户规模；业务天然适合用户车辆生命周期运营。

29. 如果要做“门店推荐 + 服务包推荐”的联合排序，你会如何平衡距离、价格、库存、评分、排队时长和利润率？
   - 来源映射：O2O 汽车服务平台的典型多目标排序问题。

30. 途虎已拓展新能源汽车维修、检测、充电桩合作等业务。新能源业务接入后，后端模型要新增哪些核心实体和流程？
   - 来源映射：公开资料提到动力电池维修、检测、充电桩销售、与整车厂合作等新能源服务。

31. 如果要评估一家门店的服务质量，你会从履约时长、返修率、投诉率、复购率、核销率等指标中如何选核心 KPI，并如何防止指标被刷？
   - 来源映射：大规模加盟/合作门店网络天然需要门店质量治理与风控。

## G. 开放题 / 方案题

32. 请你设计一个“途虎门店履约中台”，服务商品下单、库存调拨、预约排班、到店核销、售后返修五类场景，给出服务拆分与事件流。
   - 来源映射：公开业务模式高度依赖履约协同。

33. 如果让你在 90 天内优化“下单成功后到店服务完成”这一链路，你会优先改哪三件事？为什么？
   - 来源映射：适合考察业务理解、平台拆解和工程优先级。

34. 你认为汽车后市场 O2O 平台与普通电商平台在后端架构上的最大差异是什么？这些差异会如何影响数据库设计、缓存设计和接口设计？
   - 来源映射：途虎的核心差异就在于“商品交易 + 线下服务履约 + 门店网络”。

35. 如果你加入途虎做后端开发，你会优先补哪三块业务知识，才能更快和供应链、门店、运营团队对齐？
   - 来源映射：公司/岗位匹配类高频开放题。

# ORG_MAP.md

## 目标

统一 `Job-interview-Wiki` 的公司 / 事业群（或业务线）/ 部门命名口径，避免后续内容扩张时目录树漂移。

原则：
- 第一层只放公司名称
- 第二层优先放该公司在招聘语境下更稳定、更常见的事业群 / 业务线口径
- 第三层放部门、岗位归属团队或技术职能方向
- 不强行发明内部编制名，优先使用公开可识别名称

---

## 8 家公司组织映射

### 1. 字节跳动
**目录口径：按产品 / 业务线**

原因：
- 字节跳动公开招聘与候选人认知里，更容易稳定识别的是产品线或业务线
- 传统“事业群”口径不如业务线口径稳定

当前已落库：
- 抖音电商 / 商业化技术

建议扩展方向：
- 抖音主站 / 推荐架构
- 抖音生活服务 / 服务端研发
- 飞书 / 后端平台

---

### 2. 腾讯
**目录口径：按事业群缩写对应的公开常用中文名**

原因：
- 腾讯业务边界在招聘语境里常以事业群认知
- 候选人最容易识别的是微信事业群、平台与内容事业群等口径

当前已落库：
- 微信事业群 / 基础平台
- 平台与内容事业群 / 后台研发

建议扩展方向：
- 云与智慧产业事业群 / 基础架构
- 微信事业群 / 客户端平台

---

### 3. 阿里巴巴
**目录口径：按大业务集团**

原因：
- 阿里体系变化较多，但外部最稳定的认知单位通常是淘天集团、阿里云等大业务集团
- 不建议过细使用不稳定的内部子组织名作为第二层

当前已落库：
- 淘天集团 / 技术平台
- 淘天集团 / 搜广推
- 阿里云 / 基础架构

建议扩展方向：
- 本地生活 / 服务端研发
- 国际数字商业 / 技术平台

---

### 4. 美团

- 酒旅 / 服务端研发

### 5. 拼多多

- 更适合按主站、Temu、买菜、农业与供应链、平台技术等公开业务场景划分

当前已落库：

建议扩展方向：
- Temu / 服务端
- 农业与供应链 / 平台研发
- 平台与基础技术 / 基础架构
- 中后台职能系统 / 服务端研发

## 后续维护规则

1. 如果某个公司公开口径不稳定，优先选“候选人最容易识别”的业务线名称
2. 第二层不要频繁改名，新增内容优先复用已有口径
3. 第三层尽量使用技术/团队职能名，而不是临时项目名
4. 新增 question-bank 或 JSON 前，先检查是否已存在可复用目录
5. 若后续发现某公司当前第二层命名明显不合理，再统一迁移，不零散改动

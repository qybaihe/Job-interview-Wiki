# Job Interview Wiki

一个面向中文求职场景的**结构化面试知识库**，重点沉淀大厂技术岗位的：

- 结构化面经（JSON）
- 可阅读题库（Markdown）
- 公司 / 业务线 / 部门维度的分类索引

仓库目标不是简单堆“面试题合集”，而是把公开可验证的面试经验整理成一个**可持续扩展、可检索、可按方向深挖**的知识库。

---

## 仓库解决什么问题

求职资料常见有几个问题：

1. **碎片化**：大量内容散落在牛客、CSDN、博客、论坛，难以系统查找
2. **分类混乱**：公司、事业群、部门、岗位方向经常混在一起
3. **题库和面经混写**：既不利于阅读，也不利于结构化检索
4. **重复严重**：同一题目换个说法反复出现
5. **缺乏业务上下文**：只剩八股，没有方向差异

这个仓库希望把这些信息整理成两层资产：

- **面经 JSON**：保留单场面试的轮次、题目顺序、来源、感悟
- **题库 Markdown**：把高频问题抽象整理成可复习的专题题库

## 内容长什么样

### 1) 题库（Markdown）
用于复习和横向归纳，通常包含：

- 基础八股
- 算法题
- 项目深挖
- 系统设计
- 业务场景题
- 追问链路

适合：
- 面试前复习
- 同方向高频题梳理
- 不同公司同方向横向对比

### 2) 面经（JSON）
用于保留单场面试的结构化信息，通常包含：

- 公司 / 业务线 / 部门 / 岗位
- 年份与来源
- 面试轮次
- 每轮题目
- 每题 insight / 感悟
- 整场面试总结

适合：
- 检索某一方向的真实问题分布
- 抽取题库素材
- 后续做统计和自动化整理

## 目录结构

```text
companies/
  公司名称/
    事业群或业务线/
      部门或岗位方向/
        question-bank.md
        算法-question-bank.md
        客户端开发-question-bank.md
        interview-experiences/
          2024-公司-业务线-部门-岗位-面经.json
```

设计原则：

- **第一层**：公司名称
- **第二层**：事业群 / 业务线 / 核心业务板块
- **第三层**：部门 / 岗位方向 / 技术职能
- 题库与面经**并列维护**，不混写

---

## 数据设计原则

### 题库负责“抽象归纳”
题库强调：
- 去重
- 专题化
- 可阅读性
- 可复习性

### JSON 负责“结构化保真”
面经 JSON 强调：
- 保留面试轮次与上下文
- 保留来源链接
- 保留题目出现顺序
- 保留候选人的经验感悟

### 不做的事情
- 不把整篇原文直接复制进 JSON
- 不把无来源碎片当作真实面经入库
- 不把样板文件误当成真实产能
- 不收录明显涉及隐私或敏感泄露的信息

## 推荐使用方式

### 如果你是求职者
可以这样用：
- 先按公司找方向
- 再看对应题库，了解高频问题域
- 再读几份同方向 JSON 面经，建立真实面试预期

### 如果你是维护者
建议按这个顺序推进：
1. 先找完整公开面经长帖
2. 生成结构化 JSON
3. 抽取题目补题库
4. 再做去重、归类和命名清洗

## 现阶段最重要的工作

仓库现在最需要的，不再是单纯“多加几份内容”，而是：

1. **补薄弱方向**：把只有 1 份面经或只有样板的方向补到可用状态
2. **清理样板统计**：避免 README 和真实内容口径混乱
3. **统一题库结构**：让不同目录下的题库更容易横向使用
4. **强化真实性门槛**：优先公开可验证来源，避免低质量搬运
5. **建立长期 backlog**：保证自动推进时始终有高质量任务可做

# 58同城 / 平台后端 / 后端开发 question bank

> 来源说明：本题库基于 58 同城公开业务语境（分类信息 / 招聘 / 房产 / 本地生活 / 交易撮合 / 商家平台）与可审计项目规则整理，定位为**保守聚合 question-bank**。本版不冒充逐轮真实面经，后续若补到高质量公开长帖，可继续补 `interview-experiences/*.json`。
>
> 证据边界：当前主要是 `public-tech-basis-enriched-not-single-interview-post`，适合做新公司首版正式题库，不适合宣称“逐题来自单篇公开面经原帖”。

## 一、平台业务理解与项目深挖

1. 58 同城这类本地生活/分类信息平台的核心后端链路有哪些？如果让你拆模块，你会如何划分用户、内容、交易、商家、搜索和推荐服务？
2. 分类信息平台和纯电商平台相比，后端在“信息发布、审核、曝光、撮合、成交”链路上最大的差异是什么？
3. 如果同一个平台同时承载招聘、房产、二手、本地服务等多业务线，共享基础设施时最容易出现哪些平台化问题？
4. 你做过的平台型项目里，哪些能力是“业务服务”可复用的，哪些必须下沉成统一中台或基础组件？
5. 如果 58 同城某个频道在大促或节假日流量暴涨 10 倍，平台后端第一优先要保护什么：发布链路、搜索链路、商家链路，还是交易链路？为什么？
6. 如果用户发帖、商家投放、用户搜索、IM 咨询、成交转化是一个闭环，你会如何设计关键指标和观测链路？

## 二、内容发布、审核与多业务线平台化

7. 分类信息发布场景下，如何设计发帖/改帖接口，既支持高并发写入，又支持后续审核、下架、重发和风控拦截？
8. 如果房产、招聘、本地服务等频道字段模型差异很大，发布系统如何做可扩展的数据建模？是大宽表、EAV、JSON 扩展字段，还是分频道子模型？
9. 内容审核通常包含机器审核、人工审核和申诉回流，状态机应该怎么设计，才能避免“已上线又被回滚”“审核中被重复发布”之类问题？
10. 如果商家批量导入上万条信息，怎样设计异步导入、失败重试和结果回执，避免把在线数据库打挂？
11. 用户删除帖子、商家下架商品、运营屏蔽内容时，搜索索引、缓存、推荐和消息通知如何一起收敛？
12. 如果平台要支持“发布模板 + 字段校验 + 风险规则热更新”，规则引擎与业务服务边界应该怎么划？

## 三、搜索、推荐与曝光链路

13. 58 同城这类平台为什么搜索和推荐往往会与平台后端强耦合？后端要为搜索/推荐提供哪些基础能力？
14. 如果一个帖子刚发布，要求几秒内在搜索结果可见，常见的数据链路会怎么设计？MySQL、MQ、索引、缓存分别承担什么职责？
15. 搜索结果页、详情页、商家页读取的如果不是同一版本数据，会带来哪些体验问题？后端如何做版本收敛？
16. 热门城市、热门小区、热门职位等热点查询会把系统打到哪几层？你会如何设计缓存、预计算和热点隔离？
17. 如果搜索排序依赖实时行为数据，行为事件采集、异步处理和特征回流链路如何保证高可用与低延迟？
18. 用户搜索结果显示“可联系”，点进去却发现帖子已下架或商家已停业，这类展示与承接口径不一致如何治理？

## 四、商家平台、交易撮合与一致性

19. 58 同城除了 C 端发帖，还会有 B 端商家入驻、付费推广和线索分发；后端如何设计商家、门店、账号、资质之间的数据模型？
20. 如果商家修改营业状态、联系方式、投放预算和服务区域，如何保证前台展示、搜索承接和交易咨询口径一致？
21. 用户咨询商家、领取权益、下单成交可能跨多个系统，哪些地方必须做幂等？幂等键如何设计？
22. 线索类平台里“同一条线索不能重复分发、重复计费、重复消耗”怎么保证？数据库唯一约束、Redis 去重、消息幂等各适合放在哪一层？
23. 如果一次线索分发同时涉及预算扣减、商家配额、通知发送和数据埋点，如何设计本地事务与异步事件，避免分布式事务过重？
24. 当用户投诉“平台显示商家在线但实际无人响应”时，你会从状态同步、缓存延迟、消息可靠性还是商家配置问题开始排查？

## 五、缓存、数据库与消息链路

25. 58 同城这类平台为什么特别容易出现热点 Key、热点城市和热点频道？你会如何识别和治理？
26. 缓存穿透、击穿、雪崩在分类信息/搜索场景分别会怎么出现？对应治理策略怎么选？
27. 如果帖子详情页访问量远高于写入量，缓存更新是用 Cache Aside、双删，还是订阅 binlog 异步刷新？为什么？
28. 大量帖子按城市、频道、发布时间分页查询时，为什么深分页会成为问题？有哪些可落地方案？
29. 如果某个频道帖子表每天写入数千万条，分库分表时分片键怎么选？按城市、频道、用户还是时间？各自代价是什么？
30. MySQL 在这类平台型业务里通常承担哪些强一致职责？Redis 更适合承接哪些读放大或短状态场景？
31. MQ 在信息发布、审核、建索引、通知、埋点链路里通常扮演什么角色？如何保证消息不丢、不乱、不被重复消费放大业务错误？
32. 如果消息积压 2 小时，搜索仍能查到旧数据，你会怎样判断是消费者问题、下游依赖问题还是补偿机制缺失？

## 六、并发控制、稳定性与故障治理

33. 平台后端为什么比单体业务更强调限流、隔离、降级和熔断？你会把这些能力下沉到网关、SDK 还是服务治理框架？
34. 如果某个城市流量突增，如何防止它把全国流量一起拖垮？你会按城市、频道、商家还是接口维度做隔离？
35. 线上 CPU 飙升时，你怎么快速判断是热点请求、慢 SQL、GC、锁竞争，还是消息堆积引起的级联故障？
36. 如果审核服务故障，平台应该选择“全部阻塞发布”“先发后审”还是“部分频道降级”？你会怎么权衡？
37. 如何设计平台级监控体系，让你能同时看到发布成功率、审核时延、索引延迟、搜索点击率和商家响应率？
38. 当配置错误导致某个频道大量错误曝光时，如何做快速回滚、缓存失效、索引修复和用户补偿？

## 七、系统设计题

39. 设计一个分类信息发布平台，要求支持多频道差异化字段、审核流和搜索索引同步。
40. 设计一个高并发帖子详情系统，要求支持热点城市、热点帖子缓存保护与秒级状态更新。
41. 设计一个商家线索分发系统，要求支持配额控制、预算扣减、幂等计费和失败补偿。
42. 设计一个搜索承接一致性方案，避免“搜索可见、详情不可见、商家不可联系”的状态撕裂。
43. 设计一个多业务线平台统一账号与权限体系，兼顾 C 端用户、B 端商家、运营和审核人员。
44. 设计一个跨频道的实时运营看板，既要支持近实时统计，又不能影响在线主链路。

# CRON_QUALITY_PLAN.md

## 目标

把 `jobwiki-v2` 从“能跑”提升到“高质量稳定产出”。

新策略不是追求更频繁地跑，而是：
- **降低触发频率**，给单轮更充足的搜索、校验、整理、审阅时间
- **提高 worker 门槛**，强制每个 subagent 提供更完整、更可信、更可落库的候选
- **没有拿到完整有效结果时，不把失败伪装成成功**
- **本轮没拿到强结果，就明确汇报失败/部分完成，并在下一个调度窗口继续推进**

---

## 新运行策略

### 调度节奏
- **query / schedule 间隔：15 分钟一组**
- `schedule.everyMs = 900000`

### 单轮执行预算
- **整轮 orchestrator 目标执行时长：最多 12 分钟**
- `timeoutSeconds = 720`

### worker 预算
- 每个 worker 软超时目标：`480 ~ 540 秒`
- orchestrator 收口保留 `120 ~ 180 秒`

### 基本原则
1. **减少频率，增加单轮质量**
2. **不允许用 README / progress_state 的轻微修改冒充产出**
3. **必须优先拿“完整、可信、可落地”的结果**
4. **如果本轮没有强结果，就明确标记失败/部分完成，并交给下一轮继续，不得假完成**

---

## 对每个 subagent 的更高要求

每个 worker 不再只交“有点线索”的东西，而必须尽量交付**完整候选包**。

### 每个 worker 必须输出
在 `.tmp-runs/<timestamp>-<worker>/` 下至少提供：

1. `report.md`
   - 本轮处理的公司/方向
   - 搜索路径
   - 关键来源列表
   - 为什么认为这些来源可信
   - 最终建议采纳/不采纳什么

2. `sources.txt`
   - 至少列出本轮实际使用或明确排除的来源链接/出处
   - 不能只写“搜索过了”

3. 候选内容文件
   - `question-bank.md` 候选片段，或
   - `interview-experiences/*.json` 候选文件

4. `target-path.txt`
   - 明确正式落库目标路径

5. `checklist-update.md`
   - 对应 `TODO_COMPANY_CHECKLISTS.md` 的勾选建议

---

## 完整有效数据的判定标准

### question-bank 候选被视为“完整有效”需满足
- 至少新增 `>= 15` 条**非重复**问题
- 问题要能映射到公开来源，不允许纯编造
- 题目要有明确主题聚类，例如：
  - 交易链路
  - 一致性与补偿
  - 调度与资源治理
  - 推荐/搜广推
  - 大模型推理平台
- 候选内容达到可以被 orchestrator 直接审阅、筛选、合并的程度

### interview JSON 候选被视为“完整有效”需满足
- 至少包含完整岗位上下文
- 面试轮次、题目、结果、来源说明基本齐全
- 结构符合仓库模板和 schema
- 不允许只有空壳字段或极短摘要

### 治理类结果被视为“完整有效”需满足
- 明确修复一项真实资产问题
- 不能只是机械改计数
- 必须提升仓库长期可信度或可维护性

---

## 如果没有拿到完整有效数据，应该怎么做

这里要严格区分：

### 不允许做的事
- 不允许把 `partial` 包装成 `success`
- 不允许只改 `progress_state.json` 或 README 就结束
- 不允许因为“搜索到一点线索”就假装完成
- 不允许直接在同一轮里无限循环重试，导致整轮超时崩掉

### 允许且必须做的事
- worker 若未拿到完整有效候选，必须返回：
  - `partial`
  - `no-source`
  - `timeout`
  - `blocked`
- orchestrator 必须：
  1. 优先采纳已有强结果
  2. 如果没有强结果，则明确汇报失败原因
  3. 把未完成方向留给**下一轮调度继续**

### 结论
> “如果没有拿到，就不停止再执行一轮” 这件事，不能理解为在**同一轮**里无限重试；
> 正确做法是：**本轮在 12 分钟预算内尽可能提高成功率，若仍未拿到完整有效结果，则明确失败，并在下一轮 15 分钟调度继续推进。**

这样才能兼顾质量与稳定性，不会把 cron 自己拖死。

---

## Orchestrator 的新职责

orchestrator 不只是收集结果，而要负责：

1. **先检查主仓库是否已有脏内容可直接收口**
2. **把 5 个 worker 的任务切得更窄**
3. **优先要求每个 worker 处理 1 家公司 / 1~2 个方向**
4. **先验证来源，再决定是否合并**
5. **只采纳最强的 1~3 个结果**
6. **未达标结果明确淘汰，不混进正式仓库**
7. **每轮都向用户发完整中文汇报**

## 推荐的 5-worker 严格分工

### worker-A：头部公司深度补强
- 字节 / 腾讯 / 阿里 / 蚂蚁
- 每轮只选 1 家公司、1~2 个方向

### worker-B：平台与基础设施公司补强
- 美团 / 京东 / 百度 / 华为
- 优先挑业务差异明显、公开资料较多的方向

### worker-C：消费互联网第二梯队补强
- 拼多多 / 快手 / 小红书 / 哔哩哔哩 / 滴滴 / 网易 / 小米 / 微软中国
- 每轮只选最有把握的 1~2 家

### worker-D：新增公司第一组破冰
- OPPO / vivo / SHEIN / 携程 / 同程旅行 / 得物
- 目标是“新增公司正式落库”

### worker-E：新增公司第二组破冰
- BOSS直聘 / 58同城 / 商汤 / 理想汽车 / 小鹏汽车 / 蔚来 / 顺丰 / 菜鸟 / 美的 / 海尔 / 联想 / Oracle中国 / SAP中国 / 科大讯飞
- 若资料稀疏，优先做 question-bank 候选，不勉强做空壳 JSON

## 汇报要求（必须发给用户）

每轮结束必须按固定格式汇报：

- worker-A: <status> | 公司：... | 方向：... | 结果：...
- worker-B: <status> | 公司：... | 方向：... | 结果：...
- worker-C: <status> | 公司：... | 方向：... | 结果：...
- worker-D: <status> | 公司：... | 方向：... | 结果：...
- worker-E: <status> | 公司：... | 方向：... | 结果：...

并补充：
1. 本轮最终采纳了哪些结果
2. 哪些结果被放弃，原因是什么
3. 是否新增了公司 / 方向
4. 是否修了结构问题
5. commit / push 是否成功（带 commit hash）
6. 如果本轮未达标，下轮继续推进哪个公司

---

## 质量优先的核心判断

这套 cron 后续的目标不是“每轮看起来都很忙”，而是：

- 要么拿到**完整有效**的正式内容
- 要么明确说明为什么这轮没拿到
- 然后在下一个 15 分钟窗口继续推进

宁可少跑一点，也不要继续制造假完成。

## 12. 资源调度、容量治理与稳定性工程专项（新增）

> 说明：本节针对 TODO_FIRST_BATCH P1 要求，围绕阿里云基础架构后端面试中高频出现的资源调度、容量治理和稳定性工程问题，从面经真题中提炼整理。

### 12.1 资源调度与弹性伸缩

- K8s Scheduler 的调度流程？预选（Filtering）和优选（Scoring）分别做什么？
- 如何扩展 K8s Scheduler 以支持大规模集群调度？（percentageOfNodesToScore + 多 scheduler 水平扩展 + 一致性哈希分 Pod）
- 调度冲突怎么处理？乐观并发绑定失败后的重试策略？
- 弹性伸缩（HPA/VPA/CA）各自适用什么场景？HPA 基于什么指标触发？自定义指标怎么接入？
- 如果某类实例需求暴涨但单 AZ 容量不足，调度系统如何在跨 AZ 扩散、局部可用和成本约束之间做取舍？
- Pod 优先级（PriorityClass）和抢占（Preemption）机制怎么工作？什么场景下应该用抢占？

### 12.2 容量规划与资源超卖

- 云平台的容量规划一般怎么做？从哪些维度预估（计算/存储/网络/元数据）？
- 资源超卖的原理？超卖比怎么设定？Request 和 Limit 的关系？
- 超卖过高导致节点 OOM 或 CPU 争抢时，系统如何降级和驱逐？（QoS 等级：Guaranteed > Burstable > BestEffort，驱逐顺序）
- 如何通过历史数据和趋势预测做容量预警？（Prometheus + 时序预测 + 容量水位线告警）
- 资源碎片化问题怎么解决？（碎片整理/重调度 descheduler/节点池分级）

### 12.3 稳定性工程与故障治理

- 云平台核心 SLO/SLA 怎么定义？控制面和数据面的 SLO 有什么区别？
- 变更是稳定性最大的敌人——如何设计安全变更流程？（灰度发布 + 金丝雀 + 自动回滚 + 变更窗口）
- 故障注入和混沌工程在基础架构团队怎么落地？（ChaosBlade/LitmusChaos + 定期演练 + 故障半径控制）
- 如果监控指标正常但用户大量报障，你会怎么排查？（监控盲区：长尾延迟/部分节点异常/DNS 抖动/证书过期）
- 线上 CPU 突刺排查思路？（top → perf → 火焰图 → 定位热点函数/GC/锁竞争）
- 大规模集群的级联故障怎么防？（熔断/限流/舱壁隔离/故障域划分/快速失败）

### 12.4 Go 语言与云原生工程实践（面经补充）

- Go 的 GMP 调度模型？goroutine 和线程的区别？goroutine 栈大小？
- goroutine 泄漏怎么排查？（pprof + runtime.NumGoroutine + context 超时控制）
- Go 的 channel 和 sync 包怎么选？什么场景用 channel，什么场景用 Mutex？
- Operator 开发的核心模式？CRD + Controller 的 Reconcile 循环怎么保证幂等？
- client-go 的 Informer 机制？ListWatch 怎么工作？本地缓存和 etcd 的一致性怎么保证？

## 14. 云控制面与多租户异常治理清洗版

> 说明：这一节把前面偏分散的云平台问题收敛成更适合真实基础架构后端面试的框架题，重点突出控制面一致性、元数据漂移、多租户隔离、调度抖动和跨 AZ 容灾这些阿里云基础架构高频约束。

### 14.1 控制面状态漂移与回收
- 当控制面记录的资源状态与真实计算/存储/网络资源状态发生漂移时，系统如何识别、止损和回收？
- 如果删除请求已经成功返回，但底层资源并未真正释放，后续会带来哪些计费、调度和安全问题？如何收口？

### 14.2 多租户隔离与 noisy neighbor 治理
- 多租户系统里，一个租户的突发流量、异常任务或热元数据访问怎样影响其他租户？平台如何隔离？
- 如果资源隔离太保守会损失利用率，太激进又会影响稳定性，平台如何平衡超卖、隔离和成本？

### 14.3 元数据服务与审计对账
- 元数据服务为什么常常比业务数据本身更危险？如果元数据错了，平台层会发生什么连锁问题？
- 对于配置中心、注册中心、资源目录、权限配置这类元数据，如何设计审计日志、对账校验和回滚能力？

### 14.4 调度系统抖动与失败风暴
- 当调度系统在扩缩容、重试、抢占、失败补偿同时发生时，怎样避免形成任务风暴和资源震荡？
- 如果某类实例需求突然暴涨，而某个 AZ 容量不足，调度系统如何在局部可用、跨区扩散和成本约束之间做取舍？

### 14.5 跨 AZ / 多 Region 高可用收口
- 单机、多副本、多 AZ、多 Region 方案在控制面和数据面上分别意味着什么？哪些状态可以异步，哪些必须强约束？
- 当可用区级故障发生时，系统如何保证切换后的配置、权限、元数据和真实资源视图仍然一致？

### 14.6 面试回答建议
- 回答阿里云基础架构题时，优先围绕 6 个维度展开：控制面、元数据、租户隔离、调度、审计对账、容灾层级。
- 少讲抽象 CAP 口号，多讲资源生命周期、状态漂移、失败风暴、回滚与恢复链路怎样真正落到工程实现。

# 科大讯飞｜数字人平台｜实时交互与平台工程｜question-bank

> 说明：本题库基于科大讯飞官网与讯飞开放平台公开文档可映射的语音识别、在线语音合成、星火大模型、超拟人交互等能力整理。
> 当前更适合作为“数字人 / 实时交互平台工程”方向的正式 question-bank，不伪称为单场真实面试逐字题单。

## 岗位方向
- 数字人 / 虚拟人交互平台
- 实时语音交互后端
- 语音 + 大模型 + 多模态编排平台
- 数字人会话链路与平台稳定性

## 一、数字人实时交互链路

1. 如果做一个数字人实时对话系统，从用户说话到数字人开口回复，中间完整链路通常会经过哪些模块？
2. 为什么数字人场景往往比单独的 ASR 或 TTS 服务更强调“端到端首响应时间”？
3. 数字人交互里，为什么要把“识别首字延迟”“LLM 首 token 延迟”“TTS 首包延迟”“播报启动延迟”分开监控？
4. 如果用户感知数字人“听懂了，但开口特别慢”，你会优先排查链路中的哪几段？
5. 数字人系统为什么通常需要同时处理文本流、音频流，甚至还要同步动作 / 口型事件流？
6. 当 ASR 还没完全结束时，数字人能不能提前开始理解和生成？这样做的收益与风险是什么？

## 二、语音识别 / 合成在数字人中的落地

7. 数字人场景下的 ASR 和普通会议转写相比，为什么更关注实时性和打断处理，而不只是最终字错率？
8. 如果数字人需要支持“用户插话打断”，ASR、会话状态机和 TTS 播放控制要怎样协同？
9. 数字人 TTS 为什么更强调首包时延、连续性和韵律自然度，而不是只看整段音频最终生成完成时间？
10. 如果用户觉得数字人的声音“像在念稿子”，你会从文本规范化、停顿策略、重音预测还是情感参数去排查？
11. 多角色数字人场景下，多音色、多语速、多情感参数会给后端参数治理带来哪些复杂度？
12. 若一套数字人方案既支持短问答，也支持长段讲解，TTS 服务的切片、缓存和回放策略应该如何调整？

## 三、大模型 / 对话编排

13. 数字人接入大模型后，为什么不能把它简单理解为“ASR + Chat + TTS”三段串联？
14. 在数字人交互中，系统 Prompt、角色设定、知识库检索、上下文裁剪分别会怎样影响最终说话风格与回复稳定性？
15. 如果 ASR 有口语噪声、断句不稳、错别字，大模型理解会受到哪些影响？平台层怎么补救？
16. 数字人场景为什么比纯文本聊天更需要控制回复长度、语气和结构？
17. 如果数字人要支持实时流式回答，文本生成和语音合成之间应如何做流式衔接，才能减少整体等待时间？
18. 当大模型偶尔生成不适合直接播报的内容时，数字人平台应该在哪一层做安全审核、内容截断或改写？

## 四、实时会话管理与平台工程

19. 数字人平台为什么通常需要长连接或会话态管理，而不是把每次交互都做成完全无状态请求？
20. 一个数字人会话里，用户身份、历史轮次、角色参数、TTS 音色、当前播报状态、打断标志，哪些适合放在会话态里？
21. 如果同一用户同时打开多个数字人会话窗口，平台层要如何处理资源隔离、上下文一致性和配额控制？
22. 数字人平台为什么不能只按 QPS 容量规划，还要看会话持续时长、平均回复字数、音频播放时长和并发连接数？
23. 当数字人高峰时段出现“能连上但回复越来越慢”，你会如何区分是 ASR、LLM、TTS、编排层还是音视频通道的问题？
24. 如果让你设计一个科大讯飞数字人平台的可观测体系，你认为最关键的 5~8 个指标会是什么？

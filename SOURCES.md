# SOURCES.md

## 来源类型优先级

### A 级：高优先级来源
适合直接整理入库，但仍需人工清洗：
- 公开可访问的面经原文
- 个人博客中的完整面试复盘
- GitHub 上明确写出的面试记录
- 牛客网等平台中的完整长帖面经

### B 级：中优先级来源
可参考，但需要更谨慎：
- 碎片化问答贴
- 评论区零散回忆
- 多人转述整理帖
- 未明确区分岗位 / 年份 / 轮次的内容

### C 级：低优先级来源
原则上不直接入库：
- 没有上下文的短句题目
- 无法判断真伪的截图转述
- 二次搬运且没有原始来源的内容
- 明显营销/导流内容

---

## 清洗规则

### 1. 先区分题库与面经
#### 进入 question-bank.md 的内容
适合抽象沉淀的问题：
- 高频基础题
- 算法题
- 项目追问
- 系统设计题
- 行为面试题

#### 进入 interview-experiences/*.json 的内容
适合结构化保留的内容：
- 某次具体面试的题目顺序
- 每题对应感悟
- 整场面试总结
- 来源链接与补充说明

---

### 2. 去重规则
以下情况要尽量合并：
- 表述不同但本质相同的问题
- 同一题目的不同追问版本
- 同一家公司同一岗位反复出现的重复问题

保留方式：
- 题库保留抽象后的主问题
- JSON 保留原始出现方式

---

### 3. 命名规范
#### 公司名称
统一用主流正式中文名：
- 字节跳动
- 腾讯
- 阿里巴巴
- 美团
- 拼多多

#### 事业群 / 业务线
优先使用公开常见称呼，不强行发明：
- 微信事业群
- 抖音电商
- 淘天集团
- 到店事业群
- 智能云

#### 部门
尽量用招聘语境下常见叫法：
- 商业化技术
- 基础平台
- 技术平台
- 后端研发
- 服务端研发

---

### 4. JSON 整理规则
每份 JSON 至少包含：
- 公司
- 事业群
- 部门
- 岗位
- 来源
- 题目数组
- 每题感悟
- 最终感悟

不要做的事：
- 不要把整篇原文直接塞进 JSON
- 不要在 JSON 里写大段无结构散文
- 不要丢掉来源

---

### 5. 风险控制
以下内容要慎重：
- 明显带个人情绪的结论
- 无法验证的“内幕消息”
- 对面试官、部门、公司文化的绝对化判断
- 涉及隐私、泄露或敏感内部信息的内容

处理原则：
- 保留题目与感悟
- 弱化未经证实的断言
- 不收录敏感泄密信息

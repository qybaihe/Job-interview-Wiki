## 清洗版补充：基础平台稳定性与权限治理框架

> 说明：这一节把"基础题 + 平台题 + 系统设计题"里分散的问题收敛成更像微信基础平台后端真实面试会追问的框架题，重点放在服务治理、权限边界、配置变更风险、消息链路稳定性和平台接入治理。

### 1. 配置中心与变更止损
- 如果一次配置错误会同时影响多个业务方，配置中心应该怎样设计灰度、审计、回滚和生效范围控制？
- 当配置已经下发到客户端，但业务出现异常，如何判断是配置内容问题、下发生效问题还是客户端缓存问题？

### 2. 权限系统与账号边界
- 基础平台里的账号、角色、资源、操作权限如何拆分，才能避免权限模型过粗或过细？
- 如果发生越权调用、误授权或批量误操作，系统应该怎样做审计追踪、快速止损和责任回放？

### 3. 消息链路与异步一致性
- 当消息发送成功但消费延迟、重复消费、顺序错乱或下游部分失败时，平台如何保证幂等、补偿与告警闭环？
- 如果消息平台在高峰期出现堆积，是优先保核心链路、扩容、降级还是限流？判断依据是什么？

### 4. 服务治理与故障隔离
- 一个基础服务被大量业务依赖时，如何设计限流、熔断、降级、隔离和只读兜底，避免局部故障放大成平台级事故？
- 如果业务调用方式长期不规范，导致平台侧观测困难、容量失真、故障难定位，平台团队应如何推动治理收口？

### 5. 平台化推广与接入成本控制
- 一个公共能力明明技术上更优，为什么业务方仍然不愿接？平台团队如何通过 API 设计、迁移方案和默认能力降低接入阻力？
- 当平台能力快速演进时，如何处理版本兼容、历史接入方负担和新旧链路共存问题？

### 6. 面试回答建议
- 回答微信基础平台后端题时，尽量围绕：**变更风险、权限边界、消息闭环、故障隔离、接入治理** 这 5 条主线展开。
- 不要只停留在组件名词解释，要讲清楚：谁会出错、错了怎么发现、怎么止损、怎么恢复、怎么防止二次发生。


## 清洗版补充：发布、兼容与状态收敛统一治理框架

> 说明：这一节把前面分散的"配置漂移、依赖升级、权限基线、历史接入治理、消息/配置/权限状态不一致"继续向上合并，收敛成更像微信基础平台后端真实高阶追问的一组统一框架题。重点不再是单点组件，而是**发布如何安全、兼容如何收口、状态如何收敛、旧链路如何下线**。

### 1. 变更发布与多阶段止损
- 一个基础平台同时承载配置变更、SDK 升级、协议升级和权限模型调整时，怎样把"发布"拆成可灰度、可观测、可回滚的多个阶段，而不是一次性把风险打满？
- 如果线上问题只在"新配置 + 新 SDK + 某类历史客户端"组合下触发，你如何设计发布哨兵、风险开关和快速止损机制，让排障不至于陷入黑盒？

### 2. 多版本兼容与状态收敛
- 当新旧协议、新旧 SDK、新旧权限模型必须长期共存时，平台怎样定义真正需要收敛的状态主线，避免每个接入方都维护一套自己的解释口径？
- 如果配置中心看到的是版本 A、生效实例跑的是版本 B、权限校验依赖的是版本 C，平台如何判断最终以谁为准，并让状态最终收敛到单一真相？

### 3. 权限、配置与消息的一致性边界
- 为什么基础平台很多事故不是某一个组件单点失败，而是"配置变了、权限没变""权限回滚了、消息还在放""消息补发了、审计状态没跟上"这类跨系统状态错位？
- 如果必须在配置、权限、消息、审计之间划定一致性边界，你会如何区分强一致、最终一致和人工兜底区，并明确每类异常的恢复路径？

### 4. 历史接入方治理与下线节奏
- 旧业务、旧 SDK、旧调用习惯为什么会成为基础平台的长期债务池？平台团队应如何识别"必须强推迁移"的高风险接入方，以及"可以容忍但要隔离"的历史残留？
- 当平台终于要下线一批老能力时，怎样设计公告、巡检、影子流量、熔断保护和兜底回滚，避免一次"技术正确"的收口演变成大面积线上事故？

### 5. 平台团队的治理判断力
- 在基础平台场景里，为什么很多问题看起来是技术细节，实际考的是"哪个状态该统一、哪个成本该让业务承担、哪个风险必须平台兜底"的判断力？
- 如果你来负责这类平台，你会怎样向面试官说明：不是所有差异都要消灭，但所有高风险差异都必须可见、可控、可收口？

### 6. 面试回答建议
- 回答这类题时，优先围绕：**变更阶段化、兼容主线、状态真相、一致性边界、历史下线** 5 条主线展开。
- 不要继续机械罗列"配置漂移 / SDK 升级 / token 过期 / 白名单失控"这些尾词变体，而要把它们统一收进：**发布怎么控风险，状态怎么收敛，旧链路怎么退出** 的治理框架里。

## IM消息系统与存储架构专项（社招高频追问）

> 来源：牛客腾讯后台社招六面面经、知乎腾讯社招面试全流程、腾讯云微信后台架构技术文章、Go语言中文网腾讯后端面经，结合微信公开技术博客（libco协程库、Mars网络库、微信后台架构演进）交叉验证。微信作为亿级IM产品，消息系统和存储架构是WXG后端社招的独特考点。

### 1. 消息同步模型
- 读扩散（Pull）和写扩散（Push）的区别？各自的优缺点？微信采用哪种模型？为什么？
- 写扩散在大群场景（万人群）的写放大问题如何解决？（只写在线成员inbox+离线延迟同步/混合模式）
- 消息Timeline模型是什么？inbox和outbox分别存什么？如何基于序号做增量同步？
- 如何保证消息不丢？如何保证消息不重复？（写入持久化后ACK+消息ID幂等去重）
- 消息的存储库和同步库为什么要分开？各自的数据生命周期如何设计？

### 2. 长连接管理与接入层
- 亿级在线用户的长连接接入层如何设计？（多层负载均衡→接入网关→业务逻辑层）
- 心跳机制如何设计？心跳间隔怎么定？移动端省电和及时性如何平衡？
- 用户切换网络（WiFi→4G）时连接如何迁移？QUIC的Connection ID机制如何借鉴？
- 接入网关的userId→conn映射是全局共享还是本地维护？各有什么trade-off？（微信采用"半状态"设计，本地维护+广播推送）
- 弱网环境下如何保证消息到达？（客户端本地持久化+断线重连增量同步+消息ACK确认）

### 3. 自研存储与一致性
- 微信为什么没有用开源存储（MySQL/MongoDB/Cassandra）而选择自研？（数据敏感+极高可用要求+写吞吐需求）
- 分布式存储如何保证强一致性？Paxos和Raft的区别？微信选择了哪种？
- 多副本写入时如何处理脑裂？如何做自动故障切换？
- 存储层如何支持消息写扩散的极高写入吞吐？（批量写入+异步复制+WAL）
- 数据生命周期管理：消息数据保留多久？同步数据保留多久？过期数据如何清理？

### 4. 协程与高性能网络编程
- libco协程库的设计思想？为什么微信后台选择协程而非线程池？（协程切换成本远低于线程切换，适合高并发IO密集场景）
- C++20协程和libco的区别？stackful和stackless协程各有什么优缺点？
- epoll的工作原理？ET和LT模式的区别？为什么高性能服务器通常选ET？
- 如何设计一个基于UDP的可靠传输协议？需要实现哪些机制？（序列号+ACK+超时重传+滑动窗口+拥塞控制，参考QUIC）

### 5. Go语言深度（WXG后端Go方向）
- Go的GMP调度模型？G、M、P分别是什么？GOMAXPROCS的作用？
- goroutine的栈管理？初始栈大小？如何动态增长和收缩？
- channel的底层实现？有缓冲和无缓冲channel的区别？关闭channel后读写会发生什么？
- Go的GC算法？三色标记法的过程？写屏障的作用？GOGC参数如何调优？
- context包的设计？WithCancel/WithTimeout/WithValue的使用场景？context如何在goroutine间传递取消信号？
- Redis的渐进式rehash和Go map/Java HashMap的rehash有什么区别？（Redis分多次迁移避免阻塞，Go map和Java HashMap一次性rehash）

### 6. 系统设计题（IM方向）
- 设计一个支持亿级用户的IM消息系统，要求消息不丢不重、支持离线消息、支持万人群聊。
- 设计一个高可用的消息推送系统，支持在线推送和离线推送（APNs/FCM），如何处理推送风暴？
- 设计一个分布式消息存储系统，要求支持按会话查询、按时间范围查询、支持消息搜索。
- 设计一个全链路消息追踪系统，从发送方到接收方，如何追踪每条消息的状态和延迟？
